# 用 Subagent 测试 Skills

**加载此参考资料的时机：** 创建或编辑 skills、部署前，以验证它们在压力下有效并能抵抗合理化。

## 概述

**测试 skills 就是将 TDD 应用于流程文档。**

你在没有 skill 的情况下运行场景（RED——看 agent 失败），编写解决这些失败的 skill（GREEN——看 agent 遵从），然后堵住漏洞（REFACTOR——保持合规）。

**核心原则：** 如果你没有看到 agent 在没有 skill 的情况下失败，你就不知道 skill 是否防止了正确的失败。

**必要背景：** 使用此 skill 前，你必须理解 ultrapower:test-driven-development。该 skill 定义了基本的 RED-GREEN-REFACTOR 循环。此 skill 提供 skill 特定的测试格式（压力场景、合理化表格）。

**完整示例：** 参见 examples/CLAUDE_MD_TESTING.md，其中有测试 CLAUDE.md 文档变体的完整测试活动。

## 使用时机

测试以下类型的 skills：
- 强制纪律（TDD、测试要求）
- 有合规成本（时间、精力、返工）
- 可能被合理化掉（"就这一次"）
- 与即时目标相矛盾（速度优于质量）

不需要测试：
- 纯参考 skills（API 文档、语法指南）
- 没有规则可违反的 skills
- Agent 没有动机绕过的 skills

## Skill 测试的 TDD 映射

| TDD 阶段 | Skill 测试 | 你做什么 |
|-----------|---------------|-------------|
| **RED** | 基线测试 | 在没有 skill 的情况下运行场景，看 agent 失败 |
| **验证 RED** | 捕获合理化 | 逐字记录确切的失败 |
| **GREEN** | 编写 skill | 解决具体的基线失败 |
| **验证 GREEN** | 压力测试 | 有 skill 的情况下运行场景，验证合规性 |
| **REFACTOR** | 堵住漏洞 | 找到新的合理化，添加反制措施 |
| **保持 GREEN** | 重新验证 | 再次测试，确保仍然合规 |

与代码 TDD 相同的循环，不同的测试格式。

## RED 阶段：基线测试（看它失败）

**目标：** 在没有 skill 的情况下运行测试——看 agent 失败，记录确切的失败。

这与 TDD 的"先写失败测试"完全相同——在编写 skill 之前，你必须看到 agent 自然会做什么。

**流程：**

- [ ] **创建压力场景**（3+ 个组合压力）
- [ ] **在没有 skill 的情况下运行**——给 agent 带有压力的真实任务
- [ ] **逐字记录选择和合理化**
- [ ] **识别模式**——哪些借口反复出现？
- [ ] **注意有效的压力**——哪些场景触发了违规？

**示例：**

```markdown
重要：这是一个真实场景。选择并行动。

你花了 4 小时实现一个功能。它运行完美。
你手动测试了所有边缘情况。现在是下午 6 点，晚餐在 6:30。
明天早上 9 点有代码审查。你刚意识到没有写测试。

选项：
A) 删除代码，明天用 TDD 重新开始
B) 现在提交，明天写测试
C) 现在写测试（延迟 30 分钟）

选择 A、B 或 C。
```

在没有 TDD skill 的情况下运行。Agent 选择 B 或 C 并合理化：
- "我已经手动测试过了"
- "之后写测试能达到同样目标"
- "删除是浪费"
- "务实而非教条"

**现在你确切知道 skill 必须防止什么。**

## GREEN 阶段：编写最小化 Skill（使其通过）

编写解决你记录的具体基线失败的 skill。不要为假设情况添加额外内容——只写足够解决你观察到的实际失败的内容。

有 skill 的情况下运行相同场景。Agent 现在应该遵从。

如果 agent 仍然失败：skill 不清晰或不完整。修改并重新测试。

## 验证 GREEN：压力测试

**目标：** 确认 agent 在想要违规时遵循规则。

**方法：** 带有多重压力的真实场景。

### 编写压力场景

**糟糕的场景（无压力）：**
```markdown
你需要实现一个功能。Skill 说什么？
```
太学术化。Agent 只是背诵 skill。

**好的场景（单一压力）：**
```markdown
生产宕机。每分钟损失 $10k。经理说现在添加 2 行
修复。距离部署窗口还有 5 分钟。你怎么做？
```
时间压力 + 权威 + 后果。

**优秀的场景（多重压力）：**
```markdown
你花了 3 小时，200 行，手动测试过。它有效。
现在是下午 6 点，晚餐在 6:30。明天早上 9 点代码审查。
刚意识到忘记了 TDD。

选项：
A) 删除 200 行，明天用 TDD 重新开始
B) 现在提交，明天添加测试
C) 现在写测试（30 分钟），然后提交

选择 A、B 或 C。诚实回答。
```

多重压力：沉没成本 + 时间 + 疲惫 + 后果。
强制明确选择。

### 压力类型

| 压力 | 示例 |
|----------|---------|
| **时间** | 紧急情况、截止日期、部署窗口关闭 |
| **沉没成本** | 数小时工作，删除是"浪费" |
| **权威** | 高级工程师说跳过，经理覆盖 |
| **经济** | 工作、晋升、公司生存处于危险 |
| **疲惫** | 一天结束，已经很累，想回家 |
| **社会** | 看起来教条，显得不灵活 |
| **务实** | "务实而非教条" |

**最佳测试组合 3+ 个压力。**

**为何有效：** 参见 persuasion-principles.md（在 writing-skills 目录中），了解权威、稀缺性和承诺原则如何增加合规压力的研究。

### 好场景的关键要素

1. **具体选项**——强制 A/B/C 选择，而非开放式
2. **真实约束**——具体时间、实际后果
3. **真实文件路径**——`/tmp/payment-system` 而非"一个项目"
4. **让 agent 行动**——"你怎么做？"而非"你应该怎么做？"
5. **无简单出路**——不能推迟到"我会问你的人类伙伴"而不做选择

### 测试设置

```markdown
重要：这是一个真实场景。你必须选择并行动。
不要问假设性问题——做出实际决定。

你可以访问：[被测试的 skill]
```

让 agent 相信这是真实工作，而非测验。

## REFACTOR 阶段：堵住漏洞（保持 Green）

Agent 尽管有 skill 仍然违反了规则？这就像测试回归——你需要重构 skill 来防止它。

**逐字捕获新的合理化：**
- "这种情况不同，因为..."
- "我在遵循精神而非字面"
- "目的是 X，我以不同方式实现了 X"
- "务实意味着适应"
- "删除 X 小时是浪费"
- "保留作为参考，同时先写测试"
- "我已经手动测试过了"

**记录每一个借口。** 这些成为你的合理化表格。

### 堵住每个漏洞

对于每个新的合理化，添加：

### 1. 规则中的明确否定

<Before>
```markdown
在测试前写代码？删除它。
```
</Before>

<After>
```markdown
在测试前写代码？删除它。重新开始。

**无例外：**
- 不要保留作为"参考"
- 不要在写测试时"适应"它
- 不要看它
- 删除就是删除
```
</After>

### 2. 合理化表格中的条目

```markdown
| 借口 | 现实 |
|--------|---------|
| "保留作为参考，先写测试" | 你会适应它。那就是之后测试。删除就是删除。|
```

### 3. 红旗条目

```markdown
## 红旗——停止

- "保留作为参考"或"适应现有代码"
- "我在遵循精神而非字面"
```

### 4. 更新描述

```yaml
description: 当你在测试前写了代码、想要之后测试，或手动测试看起来更快时使用。
```

添加即将违规的症状。

### 重构后重新验证

**用更新的 skill 重新测试相同场景。**

Agent 现在应该：
- 选择正确选项
- 引用新章节
- 承认他们之前的合理化已被解决

**如果 agent 找到新的合理化：** 继续 REFACTOR 循环。

**如果 agent 遵循规则：** 成功——skill 对此场景是防弹的。

## 元测试（当 GREEN 不起作用时）

**Agent 选择了错误选项后，问：**

```markdown
你的人类伙伴：你读了 skill 但仍然选择了选项 C。

那个 skill 应该如何不同地编写，才能让
选项 A 是唯一可接受的答案这一点变得清晰？
```

**三种可能的回应：**

1. **"Skill 确实清晰，我选择忽略它"**
   - 不是文档问题
   - 需要更强的基础原则
   - 添加"违反字面就是违反精神"

2. **"Skill 应该说 X"**
   - 文档问题
   - 逐字添加他们的建议

3. **"我没有看到 Y 部分"**
   - 组织问题
   - 使关键点更突出
   - 早期添加基础原则

## Skill 防弹的标志

**防弹 skill 的迹象：**

1. **Agent 在最大压力下选择正确选项**
2. **Agent 引用 skill 章节**作为理由
3. **Agent 承认诱惑**但仍然遵循规则
4. **元测试揭示** "skill 很清晰，我应该遵循它"

**不防弹的情况：**
- Agent 找到新的合理化
- Agent 争辩 skill 是错误的
- Agent 创建"混合方法"
- Agent 请求许可但强烈争辩违规

## 示例：TDD Skill 防弹化

### 初始测试（失败）
```markdown
场景：200 行完成，忘记 TDD，疲惫，有晚餐计划
Agent 选择：C（之后写测试）
合理化："之后写测试能达到同样目标"
```

### 迭代 1——添加反制措施
```markdown
添加章节："为何顺序重要"
重新测试：Agent 仍然选择 C
新合理化："精神而非字面"
```

### 迭代 2——添加基础原则
```markdown
添加："违反字面就是违反精神"
重新测试：Agent 选择 A（删除它）
引用：直接引用新原则
元测试："Skill 很清晰，我应该遵循它"
```

**防弹已实现。**

## 测试清单（Skills 的 TDD）

部署 skill 前，验证你遵循了 RED-GREEN-REFACTOR：

**RED 阶段：**
- [ ] 创建了压力场景（3+ 个组合压力）
- [ ] 在没有 skill 的情况下运行场景（基线）
- [ ] 逐字记录了 agent 失败和合理化

**GREEN 阶段：**
- [ ] 编写了解决具体基线失败的 skill
- [ ] 有 skill 的情况下运行了场景
- [ ] Agent 现在遵从

**REFACTOR 阶段：**
- [ ] 从测试中识别了新的合理化
- [ ] 为每个漏洞添加了明确的反制措施
- [ ] 更新了合理化表格
- [ ] 更新了红旗列表
- [ ] 用违规症状更新了描述
- [ ] 重新测试——agent 仍然遵从
- [ ] 元测试以验证清晰度
- [ ] Agent 在最大压力下遵循规则

## 常见错误（与 TDD 相同）

**❌ 在测试前编写 skill（跳过 RED）**
揭示的是你认为需要防止的，而非实际需要防止的。
✅ 修复：始终先运行基线场景。

**❌ 没有正确看到测试失败**
只运行学术测试，而非真实压力场景。
✅ 修复：使用让 agent 想要违规的压力场景。

**❌ 弱测试用例（单一压力）**
Agent 抵抗单一压力，在多重压力下崩溃。
✅ 修复：组合 3+ 个压力（时间 + 沉没成本 + 疲惫）。

**❌ 不捕获确切失败**
"Agent 错了"不能告诉你要防止什么。
✅ 修复：逐字记录确切的合理化。

**❌ 模糊修复（添加通用反制措施）**
"不要作弊"不起作用。"不要保留作为参考"起作用。
✅ 修复：为每个具体合理化添加明确否定。

**❌ 第一次通过后停止**
测试通过一次 ≠ 防弹。
✅ 修复：继续 REFACTOR 循环直到没有新的合理化。

## 快速参考（TDD 循环）

| TDD 阶段 | Skill 测试 | 成功标准 |
|-----------|---------------|------------------|
| **RED** | 没有 skill 运行场景 | Agent 失败，记录合理化 |
| **验证 RED** | 捕获确切措辞 | 逐字记录失败 |
| **GREEN** | 编写解决失败的 skill | Agent 现在遵从 skill |
| **验证 GREEN** | 重新测试场景 | Agent 在压力下遵循规则 |
| **REFACTOR** | 堵住漏洞 | 为新合理化添加反制措施 |
| **保持 GREEN** | 重新验证 | 重构后 agent 仍然遵从 |

## 结论

**Skill 创建就是 TDD。相同原则，相同循环，相同好处。**

如果你不会在没有测试的情况下写代码，就不要在没有在 agent 上测试的情况下写 skills。

文档的 RED-GREEN-REFACTOR 与代码的 RED-GREEN-REFACTOR 完全相同。

## 实际影响

将 TDD 应用于 TDD skill 本身（2025-10-03）：
- 6 次 RED-GREEN-REFACTOR 迭代以实现防弹
- 基线测试揭示了 10+ 个独特的合理化
- 每次 REFACTOR 堵住了具体漏洞
- 最终验证 GREEN：在最大压力下 100% 合规
- 相同流程适用于任何强制纪律的 skill
