## 审查清单

**代码质量：**
- 关注点分离是否清晰？
- 错误处理是否正确？
- 类型安全（如适用）？
- 是否遵循 DRY 原则？
- 边缘情况是否已处理？

**架构：**
- 设计决策是否合理？
- 可扩展性考虑？
- 性能影响？
- 安全问题？

**测试：**
- 测试是否真正测试了逻辑（而非 mock）？
- 边缘情况是否已覆盖？
- 是否在需要的地方有集成测试？
- 所有测试是否通过？

**需求：**
- 是否满足所有计划需求？
- 实现是否与规格匹配？
- 是否有范围蔓延？
- 破坏性变更是否已记录？

**生产就绪性：**
- 迁移策略（如有 schema 变更）？
- 是否考虑了向后兼容性？
- 文档是否完整？
- 是否有明显 bug？

## 关键规则

**应该：**
- 按实际严重程度分类（不是所有问题都是严重）
- 具体说明（文件:行号，而非模糊描述）
- 解释问题为何重要
- 承认优点
- 给出明确结论

**不应该：**
- 不检查就说"看起来不错"
- 将小问题标记为严重
- 对未审查的代码给出反馈
- 含糊其辞（"改进错误处理"）
- 回避给出明确结论
