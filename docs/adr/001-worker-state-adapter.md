# ADR-001: WorkerStateAdapter 抽象层设计

**状态**: 接受

**日期**: 2026-03-05

**作者**: P1-5 Architecture Team

---

## 背景

### 问题陈述

ultrapower 中 MCP 和 Team 两个模块各自独立管理 Worker 状态：

* MCP 使用 SQLite 数据库（`job-state-db.ts`）存储任务状态

* Team 使用 JSON 文件存储心跳和注册信息

这导致：

* **代码重复**: 400-500 行重复的状态管理逻辑

* **接口不统一**: 两个模块无法互相查询 Worker 状态

* **维护成本高**: 新增 Worker 类型需要在两处实现相同逻辑

* **性能低下**: 无法跨模式优化查询和缓存

### 约束条件

* 必须保持向后兼容，现有 MCP 和 Team 功能不能中断

* 不能强制依赖 SQLite（某些环境可能不支持）

* 迁移必须分阶段进行，每阶段独立可验证

* 性能目标：健康检查从 50ms 降至 <10ms

---

## 决策

创建 `WorkerStateAdapter` 抽象层，统一 SQLite 和 JSON 两种后端实现，支持跨模式查询和性能优化。

### 选择的方案

**分层架构**:
```
┌─────────────────────────────────────┐
│  MCP / Team 业务逻辑                 │
├─────────────────────────────────────┤
│  WorkerStateAdapter 接口             │
├─────────────────────────────────────┤
│  SQLite Adapter  │  JSON Adapter     │
├─────────────────────────────────────┤
│  SQLite DB       │  JSON Files       │
└─────────────────────────────────────┘
```

**核心接口**:

* `upsert(worker)` - 插入或更新

* `get(workerId)` - 获取单个

* `list(filter)` - 列表查询

* `delete(workerId)` - 删除

* `healthCheck(workerId)` - 健康检查

* `batchUpsert(workers)` - 批量操作

* `cleanup(maxAgeMs)` - 清理过期数据

### 实施方式

**Phase 1**: 创建抽象层（不修改现有代码）

* 定义 `WorkerState` 统一数据模型

* 实现 SQLite 和 JSON 两种适配器

* 创建工厂函数支持自动回退

**Phase 2**: MCP 迁移（保留回退机制）

* 修改 `job-management.ts` 使用适配器

* 环境变量 `WORKER_BACKEND` 控制切换

* 失败时自动回退到旧实现

**Phase 3**: Team 迁移（保留 JSON 文件兼容）

* 修改 `worker-health.ts` 和 `worker-restart.ts`

* 优先查询适配器，不存在时回退 JSON

* 完全向后兼容

**Phase 4**: 性能优化

* 实现缓存层（5s TTL）

* 并发控制装饰器

* 性能基准测试

---

## 后果

### 正面影响

✅ **消除重复代码**: 减少 400-500 行重复逻辑
✅ **统一接口**: MCP 和 Team 使用同一 API
✅ **性能提升**: 4-5x 查询速度提升（缓存 + 批量操作）
✅ **易于扩展**: 新 Worker 类型接入从 2 天降至 4 小时
✅ **灵活存储**: 支持 SQLite 和 JSON，可扩展其他后端
✅ **向后兼容**: 现有代码无需修改，自动回退机制

### 负面影响

⚠️ **抽象层开销**: 多一层函数调用（可通过缓存缓解）
⚠️ **迁移复杂度**: 需要分 4 个阶段逐步迁移
⚠️ **测试成本**: 需要为两种适配器编写完整测试
⚠️ **依赖管理**: SQLite 是可选依赖，需要优雅降级

### 权衡

| 维度 | 选择 | 理由 |
| ------ | ------ | ------ |
| 性能 vs 简单性 | 性能优先 | 缓存层可消除开销，收益 4-5x |
| 统一 vs 灵活 | 统一接口 + 灵活实现 | 适配器模式支持多种后端 |
| 迁移速度 vs 稳定性 | 分阶段迁移 | 每阶段独立可验证，风险可控 |

---

## 替代方案

### 方案 A: 强制统一到 SQLite

**优点**:

* 单一数据库，查询能力强

* 实现简单，无需适配器

**缺点**:

* 某些环境可能无法安装 SQLite

* Team 模式下 JSON 文件更轻量

* 强制迁移成本高

**为什么未采用**: 不够灵活，无法适应多样化部署环境

### 方案 B: 强制统一到 JSON

**优点**:

* 无依赖，部署简单

* 文件系统兼容性好

**缺点**:

* 查询能力弱，无法高效过滤

* 性能差（逐文件扫描）

* 不适合大规模 Worker 管理

**为什么未采用**: 性能无法满足需求

### 方案 C: 保持现状（不统一）

**优点**:

* 无迁移成本

* 各模块独立演进

**缺点**:

* 代码重复持续增加

* 无法跨模式查询

* 新功能开发成本高

**为什么未采用**: 技术债务持续累积，长期成本更高
