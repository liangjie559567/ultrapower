# ultrapower v6.0.0 迁移指南

> **发布日期**: 2026-03-10
> **优先级**: P0（包含 BREAKING CHANGES）
> **升级时间**: < 5 分钟

---

## 概述

v6.0.0 是一个**安全加固版本**，修复了 3 个关键安全问题（D-05、D-06、D-07）。

### 核心变更

1. **permission-request 强制阻塞**（D-05）：失败时不再静默降级
2. **全部 15 类 HookType 严格白名单**（D-06）：未知字段统一丢弃
3. **原子写入 + 重试机制**（D-07）：所有状态文件使用 `atomicWriteJsonSyncWithRetry`

### 影响范围

- ✅ **用户无需操作**：自动升级，无配置变更
- ⚠️ **Hook 开发者**：需检查自定义 hook 输入字段
- ⚠️ **状态文件直接操作者**：必须使用原子写入 API

---

## BREAKING CHANGES

### 1. permission-request Hook 行为变更

**变更前（v5.x）**：
```typescript
// 失败时静默降级，始终返回 { continue: true }
if (result?.success === false) {
  return { continue: true }; // ❌ 安全漏洞
}
```

**变更后（v6.0.0）**：
```typescript
// 失败时强制阻塞
if (result?.success !== true) {
  return {
    continue: false,
    reason: '权限验证失败，操作已阻止',
    message: '❌ 权限验证失败，操作已阻止。'
  };
}
```

**影响**：
- `result = null/undefined` → 阻塞（之前：放行）
- `result.success = false` → 阻塞（之前：放行）
- `result.success = true` → 放行（行为不变）

**迁移操作**：
- ✅ 无需操作（自动修复安全漏洞）
- ⚠️ 如果你的 hook 依赖旧的静默降级行为，需更新逻辑

---

### 2. Hook 输入字段白名单扩展

**变更前（v5.x）**：
- 仅 4 类敏感 hook 有白名单（permission-request、setup-init、setup-maintenance、session-end）
- 其他 11 类 hook 透传未知字段

**变更后（v6.0.0）**：
- **全部 15 类 HookType** 严格白名单
- 未知字段被丢弃 + 记录安全审计日志

**影响**：
如果你的自定义 hook 向以下 HookType 传递了未定义字段，这些字段将被丢弃：

| HookType | 允许字段（完整列表见 runtime-protection.md） |
| ---------- | ---------- |
| `tool-call` | `toolName`, `toolInput`, `timestamp` |
| `tool-response` | `toolName`, `toolOutput`, `success`, `error`, `duration` |
| `agent-start` | `agent_type`, `agent_id`, `prompt` |
| `agent-stop` | `agent_id`, `success`, `output`, `duration` |
| `session-start` | `sessionId`, `directory`, `timestamp` |
| `message-sent` | `message`, `role`, `timestamp` |
| `state-change` | `mode`, `from_state`, `to_state`, `timestamp` |
| `error-occurred` | `error`, `context`, `timestamp`, `severity` |
| `file-read` | `file_path`, `success` |
| `file-write` | `file_path`, `success`, `bytes_written` |
| `custom-hook` | `hook_name`, `data` |

**迁移操作**：
1. 检查 `.omc/logs/security-audit.jsonl` 是否有 `field_filtered` 事件
2. 如有，更新 hook 代码移除未知字段

---

### 3. 状态文件写入 API 变更

**变更前（v5.x）**：
```typescript
// subagent-tracker 内部使用直接写入
fs.writeFileSync(filePath, JSON.stringify(data)); // ❌ 无原子保护
```

**变更后（v6.0.0）**：
```typescript
// 统一使用原子写入 + 重试
import { atomicWriteJsonSyncWithRetry } from './lib/atomic-write';
atomicWriteJsonSyncWithRetry(filePath, data); // ✅ 原子 + 重试
```

**影响**：
- 所有状态文件写入现在使用指数退避重试（100ms/200ms/400ms）
- Windows 平台文件锁定冲突自动重试
- 并发写入安全性提升

**迁移操作**：
- ✅ 内部自动升级，无需操作
- ⚠️ 如果你直接操作 `.omc/state/*.json`，必须使用 `atomicWriteJsonSyncWithRetry`

## 新增功能

### 1. 安全审计日志

**位置**：`.omc/logs/security-audit.jsonl`

**记录事件**：
- `field_filtered`：白名单过滤未知字段
- `permission_blocked`：permission-request 阻塞

**敏感字段自动脱敏**：
包含以下关键词的字段自动脱敏：`token`, `apiKey`, `password`, `secret`, `accessToken`, `refreshToken`, `privateKey`

**日志轮转**：
- 单文件最大：10MB
- 保留历史：7 个文件（`.1` 到 `.7`）

**示例**：
```json
{"timestamp":"2026-03-10T04:00:00Z","event":"field_filtered","data":{"hookType":"tool-call","field":"unknown"}}
{"timestamp":"2026-03-10T04:01:00Z","event":"permission_blocked","data":{"result":null}}
```

### 2. 原子写入重试机制

**新增 API**：
```typescript
atomicWriteJsonSyncWithRetry(filePath: string, data: unknown, maxRetries?: number): void
```

**重试策略**：
- 默认最多 3 次重试
- 指数退避：100ms → 200ms → 400ms
- 适用场景：高并发写入、Windows 文件锁定

## 常见问题

### Q1: 升级后 permission-request hook 总是阻塞？

**A**: 检查 hook 返回值是否明确设置 `result.success = true`。v6.0.0 采用失败优先逻辑，只有明确成功才放行。

### Q2: 自定义 hook 的字段被丢弃了？

**A**: 检查 `.omc/logs/security-audit.jsonl` 中的 `field_filtered` 事件，确认字段是否在白名单中（见 `docs/standards/runtime-protection.md` 1.1 节）。

### Q3: Windows 平台写入状态文件失败？

**A**: v6.0.0 已内置重试机制，如仍失败，检查文件是否被其他进程占用。

### Q4: 如何禁用安全审计日志？

**A**: 不推荐禁用。如需临时禁用，设置环境变量：
```bash
export OMC_DISABLE_AUDIT_LOG=1
```
