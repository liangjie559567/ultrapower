# CCG Workflow 使用指南

Claude-Codex 协作工作流（CCG Workflow）是一个自动化开发流程，通过 Claude 和 Codex 的双模型协作，确保生成的代码质量高、功能完整。

## 快速开始

### 启动工作流

```bash
/ccg-workflow
```

或指定项目类型：

```bash
/ccg-workflow new    # 新项目
/ccg-workflow old    # 老项目改造
```

系统会自动检测项目类型并路由到对应工作流。

## 工作流类型

### 新项目开发流程

适用于从零开始的项目开发。

**4 个阶段**：

1. **需求阶段** - Claude 分析需求、搜索竞品、输出技术设计
2. **开发阶段** - Codex 按周期开发，每周期输出变更摘要
3. **优化循环** - Claude 给出优化清单，Codex 按清单修改（最多 5 轮）
4. **测试循环** - Claude 给出测试清单，Codex 排查问题（最多 10 次）

**预期时间**：2 小时内完成

### 老项目改造流程

适用于现有项目的功能修改或升级。

**4 个阶段**：

1. **读取现状** - Codex 分析现有代码，输出功能流程文档
2. **生成修改文档** - Claude 根据修改需求，输出修改计划
3. **拆分开发文档** - Codex 将修改计划拆分为独立模块
4. **逐模块开发** - 每个模块完成完整闭环（优化 + 测试）

**预期时间**：单模块 < 1 小时

## 文档系统

工作流使用 7 种文档模板，存储在 `.omc/ccg/` 目录：

| 文档 | 用途 | 生成者 |
| ------ | ------ | -------- |
| `requirements.md` | 需求定义 | Claude |
| `tech-design.md` | 技术实现方案 | Claude |
| `feature-flow.md` | 功能流程（代码分析） | Codex |
| `modification-plan.md` | 修改计划（老项目） | Claude |
| `dev-modules/` | 模块开发文档 | Codex |
| `optimization-list.md` | 优化清单 | Claude |
| `test-checklist.md` | 测试清单 | Claude |

## 常见操作

### 查看当前进度

```bash
/ccg-workflow status
```

### 继续中断的工作流

如果工作流中断，重新运行命令会自动检测并询问是否继续。

### 查看生成的文档

所有文档存储在 `.omc/ccg/` 目录，可直接查看：

```bash
cat .omc/ccg/requirements.md
cat .omc/ccg/tech-design.md
```

### 手动触发优化循环

如果需要额外的优化轮次：

```bash
/ccg-workflow optimize
```

### 手动触发测试循环

如果需要额外的测试轮次：

```bash
/ccg-workflow test
```

## 工作流输出

### 新项目完成后

* ✅ 可运行的代码

* ✅ 完整的需求文档

* ✅ 技术设计文档

* ✅ 功能流程文档

* ✅ 优化清单（已完成）

* ✅ 测试报告

### 老项目完成后

* ✅ 修改后的代码

* ✅ 修改计划文档

* ✅ 各模块功能流程文档

* ✅ 优化清单（已完成）

* ✅ 测试报告

## 循环限制

为防止无限循环，工作流设置了以下限制：

* **优化循环**：最多 5 轮

* **测试循环**：最多 10 次

如果达到限制，系统会提示需要人工介入。

## 微服务项目

对于微服务架构项目，工作流会：

1. 自动检测各个服务
2. 为每个服务生成独立的功能流程文档
3. 为每个服务生成独立的修改计划
4. 逐个服务完成开发闭环

## 降级策略

如果 Codex API 不可用，系统会自动降级到 Claude 单模型，保留所有流程和提示词。

## 常见问题

### Q: 工作流中断了怎么办？

A: 重新运行 `/ccg-workflow` 命令，系统会自动检测中断点并询问是否继续。

### Q: 如何跳过某个阶段？

A: 不建议跳过。每个阶段都是必要的。如需特殊处理，请手动编辑对应的文档。

### Q: 优化循环没有收敛怎么办？

A: 达到 5 轮限制后，系统会提示。此时需要人工审查优化清单，确认是否需要继续。

### Q: 如何处理微服务之间的依赖？

A: 工作流会自动生成依赖图。在修改计划中会标注跨服务依赖，需要按依赖顺序开发。

### Q: 测试循环失败了怎么办？

A: 查看测试报告中的失败原因，手动修复代码后，重新运行 `/ccg-workflow test`。

## 最佳实践

1. **准备充分的需求** - 需求越清晰，生成的代码质量越高
2. **定期检查文档** - 每个阶段完成后，检查生成的文档是否符合预期
3. **及时反馈** - 如果发现问题，立即反馈给 Claude，不要等到最后
4. **保持代码整洁** - 在开发过程中保持代码风格一致
5. **充分测试** - 不要跳过测试循环，确保代码质量
