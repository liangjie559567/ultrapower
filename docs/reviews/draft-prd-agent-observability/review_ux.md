# UX Review: Draft PRD - Agent 可观测性平台

## 1. Flow Analysis (流程分析)

**F1 /ax-observe 命令结构**

PRD 设计了四个子命令：`--session`、`--agent`、`--tool`、`--cost`。问题在于这四个维度是正交的，用户实际需求往往是组合查询（"这次 session 里哪个 agent 花费最高"），但当前设计无法表达这种组合。

* 现状：`/ax-observe --cost` 和 `/ax-observe --agent` 是独立命令，无法组合

* 建议：改为主命令 + 过滤参数模式：`/ax-observe [--session <id>] [--by agent | tool | cost]`，支持组合

**F2 /ax-status 集成**

"可观测性摘要"区块的触发时机不明确——是每次执行 `/ax-status` 都显示，还是只在有异常时显示？

* 问题：如果默认展示，会增加 `/ax-status` 的输出噪音；如果默认隐藏，用户不知道这个功能存在

* 建议：默认显示单行摘要 `[observe] last run: 3 agents, 12.4s, ~$0.08`，`--verbose` 展开详情

**F3 用户故事与命令的映射断层**

用户故事"查看最近一次 ultrawork 执行中每个 agent 的耗时分布"需要几步才能完成：先找到 session id，再用 `--session` 过滤，再用 `--agent` 查看。PRD 没有提供"最近一次执行"的快捷入口。

* 建议：增加 `/ax-observe --last` 作为最高频操作的零参数快捷方式

---

## 2. UI/Visual Feedback (视觉反馈)

**耗时分布的 CLI 呈现**

PRD 提到"耗时分布"但未定义输出格式。在 CLI 环境下，表格和图表的选择直接影响可读性。

* 问题：如果输出原始数字列表，用户无法快速识别瓶颈

* 建议：使用对齐表格 + 相对比例条，例如：
  ```
  agent          time    share
  executor       8.2s    ████████░░  66%
  verifier       3.1s    ███░░░░░░░  25%
  explore        1.1s    █░░░░░░░░░   9%
  ```

**成本输出的精度问题**

`--cost` 输出的数字精度未定义。`$0.001234` 和 `$0.00` 都会让用户困惑。

* 建议：统一显示两位小数美元，附带 token 数：`$0.08 (input: 12,400 / output: 3,200 tokens)`

**工具调用最慢的识别**

用户故事"知道哪个工具调用最慢"需要一个明确的视觉标记，而不是让用户自己从列表中找。

* 建议：最慢的条目自动标注 `[slowest]`，超过阈值（如 5s）标注 `[warn]`

---

## 3. Usability Score (1-10)

**评分：5/10**

Reasoning：命令结构设计存在根本性的正交维度问题，四个子命令无法组合使用，导致最常见的"跨维度查询"场景需要多次执行命令并手动关联结果。用户故事与实际命令设计之间存在明显的映射断层。输出格式完全未定义，在 CLI 环境下这是高风险遗漏——同样的数据用表格和用 JSON 输出，可用性差距极大。`/ax-status` 集成的触发逻辑不清晰，可能造成输出噪音或功能不可发现两个极端。

---

## Conclusion (结论)

* Optimizable

* Top 3 Changes Needed (亟待改进):
  1. 重构子命令为组合过滤参数模式，支持 `--by agent | tool | cost` 与 `--session` 组合使用，消除正交维度断层
  2. 明确定义 CLI 输出格式：耗时用对齐表格+比例条，成本用"$X.XX (N tokens)"格式，最慢条目自动标注
  3. 增加 `/ax-observe --last` 零参数快捷入口，覆盖最高频的"查看上次执行"场景
