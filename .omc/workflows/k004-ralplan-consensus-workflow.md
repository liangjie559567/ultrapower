# Ralplan共识流程

## 背景
复杂的技术债务修复、大型重构或架构变更需要多角度评审。单一规划者可能遗漏架构缺陷或数据不一致。Ralplan通过Planner、Architect、Critic三角评审确保计划质量。

## 实现步骤

1. **Planner创建初始计划（v1.0）**
   - 分析任务范围和依赖
   - 生成详细执行计划
   - 估算工作量和风险

2. **Architect并行评审**
   - 检查架构合理性
   - 验证模块边界和接口设计
   - 评分7-10分（通过）或<7分（拒绝）

3. **Critic并行评审**
   - 检查质量标准和最佳实践
   - 验证测试覆盖和验证策略
   - 通过/拒绝决策

4. **反馈整合**
   - 如Critic拒绝，Planner根据反馈修订v2.0
   - 最多迭代5次直至达成共识
   - 所有评审者同意后进入执行

## 验证方法

- 检查计划版本历史：v1.0 → v2.0 → ... → final
- 验证所有评审者签名（Architect评分 + Critic通过）
- 确认迭代次数≤5次
- 执行后对比实际vs计划工作量

## 注意事项

- **数据一致性**：Architect常发现审计报告过时的问题
- **架构缺陷**：Critic可能识别出设计中的风险
- **迭代上限**：5次迭代后若未达成共识，需重新分析需求
- **并行评审**：Architect和Critic同时进行，加快反馈周期
