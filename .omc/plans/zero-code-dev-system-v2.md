## 1. 项目概览

### 1.1 项目目标

构建一个让零编程基础用户从模糊想法到生产级应用的完整 AI 辅助系统，支持 5 种平台类型（Web/Mobile/API/插件/桌面），具备自主学习、自主进化和自主调用能力。

### 1.2 核心能力

- **智能需求澄清**: 通过对话引擎将模糊想法转化为结构化需求
- **自动技术选型**: 基于需求自动生成最优技术方案
- **任务智能拆解**: 自动分析依赖关系并生成执行计划
- **多平台脚手架**: 一键生成 5 种平台的项目结构
- **自适应代码生成**: 根据上下文智能生成高质量代码
- **服务自动集成**: 自动集成第三方 API 和服务
- **全自动质量门禁**: 性能/安全/测试全方位自动检查
- **一键部署监控**: 自动化部署和实时监控
- **开源项目研究**: 自动分析和二次开发开源项目

### 1.3 技术约束

- 基于现有 49 agents + 71 skills + Axiom 工作流
- 遵循 4 个门禁：Expert Gate、User Gate、CI Gate、Scope Gate
- TypeScript strict mode
- 测试覆盖率 ≥ 80%
- 所有状态存储在 `.omc/` 目录

### 1.4 修订后的工时估算（解决 Critic P0-1）

## 2. Sprint 规划（7 个 Sprint，含 PoC）

### Sprint 0: PoC 技术可行性验证（Week 1-2）【新增】

**目标**: 验证核心技术假设，降低项目风险（解决 Critic P0-2）

**验证目标**:
1. AI 代码生成质量是否满足生产要求
2. 多 Agent 协作延迟是否可接受（<10s）
3. 分布式架构是否支持 1000+ 并发
4. 第三方服务集成可行性

**关键交付物**:
1. PoC 原型（3 个核心场景）
2. 性能基准测试报告
3. 技术风险评估报告
4. Go/No-Go 决策文档

**PoC 场景**:

**场景 1: 需求澄清 → 代码生成**
- 输入：用户描述"我想做一个待办事项应用"
- 输出：生成可运行的 React + TypeScript 代码
- 验收：代码可编译、可运行、符合最佳实践
- 性能目标：<10s

**场景 2: 多 Agent 协作**
- 测试：analyst → planner → executor 协作流程
- 验收：Agent 间通信正常、状态同步无误
- 性能目标：3 个 Agent 协作 <30s

**场景 3: 并发负载**
- 测试：100 个并发请求（模拟 1000+ 用户的 10% 负载）
- 验收：无请求失败、响应时间 <15s（P95）
- 架构：API 网关 + 3 个 Agent 实例

**技术决策**:
- 使用 BullMQ 作为消息队列
- 使用 Redis 作为状态存储
- 使用 PostgreSQL 作为持久化数据库
- 使用 Nginx 作为 API 网关

**验收标准**:
- [ ] 3 个 PoC 场景全部通过
- [ ] 性能基准达标（需求澄清 <10s，完整流程 <5min）
- [ ] 技术风险评估完成（识别 ≥5 个风险）
- [ ] Go/No-Go 决策通过（技术可行性确认）

**风险**:
- 风险：PoC 失败导致项目无法继续
- 缓解：提前准备备选方案（降级为半自动化）
- 监控指标：PoC 通过率、性能达标率

**工时**: 80h（设计 16h + 开发 40h + 测试 16h + 文档 8h）

**关键交付物**:

2. 需求澄清对话引擎（支持多轮对话、上下文理解）
3. 需求结构化存储机制（`.omc/zerodev/requirements/`）

- 使用 `analyst` (opus) 作为需求澄清的核心 agent
- 引入新的 `requirement-clarifier` agent（基于 analyst 扩展）
- 需求存储格式：JSON Schema + Markdown 混合格式
- API 网关：Nginx + rate limiting（100 req/min per user）

**Agent 集成路径**（解决 Architect P0-3）:

```typescript
// requirement-clarifier agent 集成接口
interface RequirementClarifierAgent {
  // 继承自 analyst agent
  extends: 'analyst';

  // 新增能力
  capabilities: {
    multiRoundDialogue: true;
    contextMemory: true;
    platformDetection: true;
  };

  // 集成方式
  integration: {
    // 1. 注册到 agent registry
    registry: 'src/agents/definitions.ts';

    // 2. 实现 agent 接口
    interface: 'src/agents/base/AgentInterface.ts';

    // 3. 添加系统提示词
    systemPrompt: 'src/agents/prompts/requirement-clarifier.md';

    // 4. 配置路由规则
    routing: {
      trigger: ['clarify', 'requirements', 'what do you need'],
      priority: 'high'
    };
  };

  // 状态管理
  state: {
    storage: '.omc/zerodev/state/requirement-clarifier-state.json';
    schema: 'src/schemas/requirement-clarifier-state.schema.json';
  };
}
```

**验收标准**:
- [ ] 用户可通过自然语言描述想法

- [ ] 自动提问澄清模糊需求（至少 3 轮对话）

- [ ] 单元测试覆盖率 ≥ 80%
- [ ] 性能：需求澄清响应时间 <10s（解决 Architect P0-1）

**风险与缓解**（解决 Critic P0-5）:
- **风险 1**: 需求理解准确度不足（影响：高，概率：中）
  - 缓解措施：引入需求确认环节（User Gate）+ 多轮对话验证
  - 监控指标：需求理解准确率（目标 ≥85%）

## 5. 质量保证策略（解决 Critic P0-4）

### 5.1 分层测试策略

**L1: 单元测试**
- 覆盖率：≥80%
- 工具：Jest + ts-jest
- 运行频率：每次提交
- 执行时间：<2min

**L2: 集成测试**
- 覆盖率：关键路径 100%
- 工具：Supertest + TestContainers
- 运行频率：每次 PR
- 执行时间：<10min

**L3: 端到端测试**
- 场景：5 种平台 × 3 个典型用例 = 15 个场景
- 工具：Playwright
- 运行频率：每日
- 执行时间：<30min

**L4: 性能测试**
- 负载：1000 并发用户
- 工具：k6
- 运行频率：每周
- 执行时间：<1h

**L5: 安全测试**
- 扫描：OWASP Top 10
- 工具：Snyk + OWASP ZAP
- 运行频率：每次发布
- 执行时间：<15min

### 5.2 自动化质量门禁

**门禁 1: Expert Gate**
- 触发：所有新功能需求
- 检查：需求完整性、技术可行性
- 自动化：需求模板校验 + AI 评审

**门禁 2: User Gate**
- 触发：PRD 终稿生成
- 检查：用户确认
- 自动化：交互式确认界面

**门禁 3: CI Gate**
- 触发：代码提交
- 检查：编译 + 测试 + 覆盖率
- 自动化：GitHub Actions
- 失败策略：阻止合并

**门禁 4: Scope Gate**
- 触发：文件修改
- 检查：是否在 manifest 范围内
- 自动化：文件路径校验
- 越界策略：警告 + 需用户确认

### 5.3 质量指标监控

| 指标 | 目标 | 监控频率 | 告警阈值 |
|-----|------|---------|---------|
| 测试覆盖率 | ≥80% | 每次提交 | <75% |
| 构建成功率 | ≥95% | 每次构建 | <90% |
| 部署成功率 | ≥95% | 每次部署 | <90% |
| 安全漏洞数 | 0 (高危) | 每日 | >0 |
| 性能回归 | <10% | 每周 | >15% |
| 用户满意度 | ≥85% | 每月 | <80% |
