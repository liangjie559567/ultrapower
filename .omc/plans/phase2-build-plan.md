## 1. 当前构建流程分析

### 1.1 现状概览

**构建命令链**: `tsc && node scripts/parallel-build.mjs && npm run compose-docs`

**并行构建任务** (parallel-build.mjs):
1. `build-skill-bridge.mjs` - 打包 skill-bridge.cjs
2. `build-mcp-server.mjs` - 打包 MCP 服务器
3. `build-codex-server.mjs` - 打包 Codex 服务器（嵌入 50 个 agent prompts）
4. `build-gemini-server.mjs` - 打包 Gemini 服务器（嵌入 50 个 agent prompts）
5. `build-bridge-entry.mjs` - 打包 team-bridge.cjs

**构建工具栈**:
- TypeScript 编译器 (tsc)
- esbuild (用于 bundle 打包)
- 自定义脚本 (文档组合、元数据同步)

### 1.2 已启用的优化

✅ **增量编译已启用**:
```json
{
  "incremental": true,
  "tsBuildInfoFile": ".tsbuildinfo"
}
```

✅ **并行构建已实现**: 5 个 esbuild 任务并行执行

✅ **跳过类型检查**: `skipLibCheck: true` 减少外部依赖类型检查

### 1.3 性能基线

根据构建输出观察:
- **tsc 编译**: 主要耗时阶段（首次构建 ~10-15s，增量 ~2-5s）
- **并行打包**: ~3-5s（5 个 esbuild 任务并行）
- **文档组合**: ~1-2s

**总耗时估算**:
- 首次完整构建: ~15-20s
- 增量构建: ~5-10s

## 2. 优化机会分析

### 2.1 TypeScript 配置优化 ⚡

**当前配置评估**: 已经相当优化，但仍有微调空间

**优化建议**:

1. **启用项目引用** (可选，适用于大型项目):
   - 当前单体配置适合中型项目
   - 如果未来模块化（如分离 CLI/Core/MCP），可考虑 project references

2. **调整 sourceMap 策略**:
   ```json
   "sourceMap": true,           // 当前: 完整 source map
   "inlineSourceMap": false,    // 建议: 保持分离以加快编译
   "inlineSources": false       // 建议: 不内联源码
   ```

3. **优化声明文件生成**:
   ```json
   "declarationMap": true       // 已启用，保持
   ```

**预期收益**: 微小（<5% 编译时间）

### 2.2 缓存策略优化 💾

**当前状态**:
- ✅ tsc 增量缓存: `.tsbuildinfo`
- ❌ esbuild 缓存: 未启用
- ❌ 依赖缓存: 未实现

**优化方案**:

1. **esbuild 元文件缓存**:
   ```javascript
   await esbuild.build({
     // ... existing config
     metafile: true,
     logLevel: 'warning'
   });
   ```

2. **智能缓存失效**:
   - 检测源文件 mtime
   - 跳过未变更的 bundle 任务

**预期收益**: 中等（增量构建提速 20-30%）

### 2.3 并行化增强 🚀

**当前并行度**: 5 个 esbuild 任务

**优化方案**:

1. **tsc + esbuild 并行**:
   - 当前: `tsc` → `parallel-build.mjs`（串行）
   - 优化: 部分 esbuild 任务可与 tsc 并行（如果不依赖 dist/ 输出）

2. **文档组合并行**:
   - 当前: 串行执行
   - 优化: 与最后的 esbuild 任务并行

**预期收益**: 小（首次构建提速 10-15%）

## 4. 风险评估与依赖

### 4.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 缓存失效逻辑错误导致构建不完整 | 高 | 中 | 添加完整性校验；保留强制完整构建选项 |
| 并行化导致竞态条件 | 中 | 低 | 仔细分析依赖关系；充分测试 |
| CI 环境缓存不兼容 | 低 | 中 | 提供缓存清理命令；文档说明 |
| 性能提升不明显 | 低 | 低 | 已有增量编译，收益可能有限 |

### 4.2 依赖项

**外部依赖**:
- ✅ esbuild (已安装)
- ✅ TypeScript (已配置增量编译)
- ✅ Node.js fs/path API

**内部依赖**:
- 需要理解现有 5 个构建脚本的输入输出
- 需要确认 dist/ 目录的生成顺序

**阻塞因素**:
- 无硬性阻塞
- 建议在 Phase 1 (测试覆盖率) 之后执行，避免并行修改构建流程

## 6. 实施建议

### 6.1 优先级排序

1. **高优先级**: 智能缓存检测（Task 3.1-3.3）
   - 收益最大，风险可控
   - 对现有流程改动最小

2. **中优先级**: 并行化优化（Task 3.4-3.6）
   - 需要仔细分析依赖
   - 收益相对较小

3. **低优先级**: TypeScript 配置微调
   - 收益极小，可暂缓

### 6.2 回滚策略

保留原始构建脚本作为备份：
```bash
npm run build:legacy  # 使用旧流程
npm run build         # 使用优化流程
```

### 6.3 后续优化方向

- **长期**: 考虑 Turborepo/Nx 等 monorepo 工具（如果项目规模扩大）
- **长期**: 探索 swc 替代 tsc（更快的编译器）
- **中期**: 实现分布式缓存（团队共享构建缓存）

## 7. 总结

**核心策略**: 在已有增量编译基础上，通过智能缓存和选择性并行化进一步提升构建速度。

**关键收益**:
- 增量构建提速 30-70%
- 开发体验改善
- 零破坏性变更

**执行建议**: 分阶段实施，优先缓存优化，充分测试后再进行并行化改造。
