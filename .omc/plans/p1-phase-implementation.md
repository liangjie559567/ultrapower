# P1 阶段实施计划

**基于**: v5.5.18 反思报告
**时间范围**: 3 个月
**优先级**: P1（高价值优化）

---

## 目标 1: 反思报告自动生成流程

### 背景
当前反思报告由 analyst agent 手动生成，耗时约 30 分钟。自动化可节省 80% 时间。

### 实施步骤

#### 1.1 数据收集模块
- **输入**: `.omc/axiom/active_context.md`（会话状态）
- **输出**: 结构化会话数据（JSON）
- **关键字段**:
  - 任务列表（完成/失败/阻塞）
  - 耗时分布
  - Agent 使用统计
  - 自动修复次数

#### 1.2 模式识别模块
- **算法**: 基于规则的模式匹配
- **识别目标**:
  - 成功模式（如 Team 并行执行）
  - 失败模式（如依赖死锁）
  - 效率瓶颈（如重复等待）

#### 1.3 报告生成模块
- **模板**: `reflection-template.md`
- **章节**:
  1. What Went Well
  2. What Could Improve
  3. Learnings
  4. Action Items
  5. 可复用模式
  6. 量化指标对比

#### 1.4 集成点
- **触发时机**: 状态从 EXECUTING → ARCHIVING
- **Hook**: `axiom-boot` 检测状态变化
- **输出**: `.omc/axiom/reflection_[timestamp].md`

### 验收标准
- [ ] 自动生成的报告包含所有 8 个章节
- [ ] 量化指标准确率 >95%
- [ ] 生成时间 <5 分钟

## 目标 2: 最佳实践知识库与工作流推荐

### 背景
当前工作流选择依赖用户经验。自动推荐可提升新手效率 50%。

### 实施步骤

#### 2.1 知识库结构
```
.omc/axiom/knowledge/
├── workflow_patterns.md（已存在）
├── agent_routing.md（已存在）
├── best_practices/
│   ├── team_mode.md
│   ├── parallel_execution.md
│   ├── security_fixes.md
│   └── performance_optimization.md
└── recommendations/
    └── workflow_recommender.json
```

#### 2.2 推荐引擎
- **输入**: 用户意图（如 "修复 6 个 P0 问题"）
- **分析维度**:
  - 任务数量（1 vs 3+ vs 10+）
  - 任务类型（bug-fix vs feature vs refactor）
  - 依赖关系（独立 vs 顺序 vs 复杂）
- **输出**: 推荐工作流 + 置信度

#### 2.3 集成到 next-step-router
- **增强**: 在路由决策时查询知识库
- **展示**: 在 AskUserQuestion 中附加 "为什么推荐" 理由

#### 2.4 反馈循环
- **收集**: 用户选择的工作流 + 实际效果
- **更新**: 调整推荐权重（成功 +1，失败 -1）

### 验收标准
- [ ] 推荐准确率 >80%（基于历史数据验证）
- [ ] 响应时间 <1 秒
- [ ] 知识库包含 10+ 个最佳实践

## 实施优先级

### Phase 1（第 1 个月）
1. 创建知识库结构（`best_practices/`）
2. 迁移现有模式到知识库
3. 开发推荐引擎原型

### Phase 2（第 2 个月）
1. 集成推荐引擎到 next-step-router
2. 开发数据收集模块
3. 创建反思报告模板

### Phase 3（第 3 个月）
1. 开发模式识别模块
2. 集成报告生成到 axiom-boot
3. 建立反馈循环

---

## 风险与缓解

### 风险 1: 自动生成报告质量不足
- **缓解**: 保留人工审查步骤，自动生成作为草稿

### 风险 2: 推荐引擎过拟合历史数据
- **缓解**: 定期重新训练，引入多样性惩罚

### 风险 3: 知识库维护成本高
- **缓解**: 使用 Markdown 格式，支持 Git 版本控制
