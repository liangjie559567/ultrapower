## 1. 项目概览

### 1.1 项目目标

构建一个让零编程基础用户从模糊想法到生产级应用的完整 AI 辅助系统，支持 5 种平台类型（Web/Mobile/API/插件/桌面），具备自主学习、自主进化和自主调用能力。

### 1.2 核心能力

- **智能需求澄清**: 通过对话引擎将模糊想法转化为结构化需求
- **自动技术选型**: 基于需求自动生成最优技术方案
- **任务智能拆解**: 自动分析依赖关系并生成执行计划
- **多平台脚手架**: 一键生成 5 种平台的项目结构
- **自适应代码生成**: 根据上下文智能生成高质量代码
- **服务自动集成**: 自动集成第三方 API 和服务
- **全自动质量门禁**: 性能/安全/测试全方位自动检查
- **一键部署监控**: 自动化部署和实时监控
- **开源项目研究**: 自动分析和二次开发开源项目

### 1.3 技术约束

- 基于现有 49 agents + 71 skills + Axiom 工作流
- 遵循 4 个门禁：Expert Gate、User Gate、CI Gate、Scope Gate
- TypeScript strict mode
- 测试覆盖率 ≥ 80%
- 所有状态存储在 `.omc/` 目录

### 1.4 修订后的工时估算（v2.1 修正）

## 2. Sprint 规划（7 个 Sprint，含 PoC）

### Sprint 0: PoC 技术可行性验证（Week 1-2）【v2.1 增强】

**目标**: 验证核心技术假设，降低项目风险

**验证目标**:
1. AI 代码生成质量是否满足生产要求
2. 多 Agent 协作延迟是否可接受（<10s）
3. 分布式架构是否支持 1000+ 并发
4. 第三方服务集成可行性

**关键交付物**:
1. PoC 原型（3 个核心场景）
2. 性能基准测试报告
3. 技术风险评估报告
4. Go/No-Go 决策文档

**PoC 场景**:

**场景 1: 需求澄清 → 代码生成**
- 输入：用户描述"我想做一个待办事项应用"
- 输出：生成可运行的 React + TypeScript 代码
- 验收：代码可编译、可运行、符合最佳实践
- 性能目标：<10s

**场景 2: 多 Agent 协作**
- 测试：analyst → planner → executor 协作流程
- 验收：Agent 间通信正常、状态同步无误
- 性能目标：3 个 Agent 协作 <30s

**场景 3: 并发负载**
- 测试：100 个并发请求（模拟 1000+ 用户的 10% 负载）
- 验收：无请求失败、响应时间 <15s（P95）
- 架构：API 网关 + 3 个 Agent 实例

**Go/No-Go 决策矩阵（v2.1 新增）**:

| 验证项 | 通过标准 | 失败后果 | Plan B |
|--------|---------|---------|--------|
| 代码生成质量 | ≥85 分 | 项目暂停 | 降级为模板库 + 人工审查 |
| 多 Agent 延迟 | <30s | 继续但调整目标 | 串行执行 + 缓存优化 |
| 并发性能 | 100 并发无失败 | 继续但降低规模 | 目标降为 100 并发用户 |
| 服务集成 | ≥3 个成功 | 继续但减少范围 | 仅支持核心 5 个服务 |

**三级退出策略（v2.1 新增）**:

**🟢 绿灯（全部通过）**：按原计划进入 Sprint 1
- 所有 4 个验证项达标
- 性能基准满足目标
- 技术风险可控

**🟡 黄灯（部分通过）**：调整目标后继续
- 代码质量 75-85 分：增加人工审查环节
- 延迟 30-60s：调整性能目标为 <60s
- 并发 50-100：目标降为 500 并发用户
- 服务集成 1-2 个：减少集成范围

**🔴 红灯（核心失败）**：项目暂停/转型
- 代码质量 <75 分：转为半自动化系统（AI 辅助 + 人工编码）
- 延迟 >60s：转为异步任务模式（用户提交后邮件通知）
- 并发 <50：转为单租户部署模式
- 服务集成 0 个：暂停项目，重新评估技术方案

**止损点（v2.1 新增）**:
- ⏱️ **时间止损**：PoC 超过 3 周仍未通过 → 项目暂停
- 💰 **成本止损**：PoC 成本超过 $20,000 → 重新评估
- 🔬 **技术止损**：核心技术假设被证伪 → 项目转型

**验收标准**:
- [ ] 3 个 PoC 场景全部通过
- [ ] 性能基准达标（需求澄清 <10s，完整流程 <5min）
- [ ] 技术风险评估完成（识别 ≥5 个风险）
- [ ] Go/No-Go 决策通过（技术可行性确认）

**工时**: 160h（设计 32h + 开发 80h + 测试 32h + 文档 16h）

### 3.2 Agent 池扩缩容策略（v2.1 新增）

**扩容触发条件**:

| 指标 | 扩容阈值 | 持续时间 | 扩容动作 | 目标状态 |
|-----|---------|---------|---------|---------|
| CPU 使用率 | >70% | 5min | +2 Worker 实例 | CPU <60% |
| 队列长度 | >50 | 3min | +3 Worker 实例 | 队列 <30 |
| P95 延迟 | >15s | 5min | +2 Worker 实例 | 延迟 <10s |
| 并发请求数 | >800 | 1min | +5 Worker 实例 | 并发能力 >1000 |

**缩容触发条件**:

| 指标 | 缩容阈值 | 持续时间 | 缩容动作 | 最小保留 |
|-----|---------|---------|---------|---------|
| CPU 使用率 | <30% | 15min | -1 Worker 实例 | 最少 3 个/池 |
| 队列长度 | <10 | 10min | -2 Worker 实例 | 最少 3 个/池 |
| 并发请求数 | <200 | 10min | -3 Worker 实例 | 最少 3 个/池 |

**扩缩容配置**:

```typescript
// Agent 池扩缩容配置
const scalingConfig = {
  // 全局限制
  minInstances: 3,           // 每个池最少 3 个实例
  maxInstances: 50,          // 全局最多 50 个实例
  
  // 冷却时间（防止频繁扩缩容）
  scaleUpCooldown: 300,      // 扩容后 5min 内不再扩容
  scaleDownCooldown: 900,    // 缩容后 15min 内不再缩容
  
  // 监控指标阈值
  metrics: {
    cpu: { 
      scaleUp: 70,           // CPU >70% 触发扩容
      scaleDown: 30          // CPU <30% 触发缩容
    },
    queueLength: { 
      scaleUp: 50,           // 队列 >50 触发扩容
      scaleDown: 10          // 队列 <10 触发缩容
    },
    latencyP95: { 
      scaleUp: 15000,        // P95 延迟 >15s 触发扩容
      scaleDown: 5000        // P95 延迟 <5s 触发缩容
    }
  },
  
  // 分池配置
  pools: {
    analyst: { min: 3, max: 15, priority: 'high' },
    planner: { min: 3, max: 10, priority: 'high' },
    executor: { min: 3, max: 25, priority: 'critical' }
  }
};
```

**三级降级机制（v2.1 新增）**:

**L1 降级（轻度过载）**:
- 触发条件：队列长度 >100
- 降级动作：启用请求限流（100 req/min → 50 req/min）
- 用户体验：响应变慢，但功能完整
- 恢复条件：队列长度 <50 持续 5min

**L2 降级（中度过载）**:
- 触发条件：队列长度 >200 或 P95 延迟 >30s
- 降级动作：禁用非核心功能（开源研究、高级优化）
- 用户体验：仅保留核心流程（需求澄清 → 代码生成 → 部署）
- 恢复条件：队列长度 <100 且延迟 <20s 持续 10min

**L3 降级（重度过载）**:
- 触发条件：队列长度 >500 或系统 CPU >90%
- 降级动作：拒绝新请求，返回 503 + 预计恢复时间
- 用户体验：系统不可用，显示维护页面
- 恢复条件：队列长度 <200 且 CPU <70% 持续 15min

### 11.2 P1-PoC：PoC 失败策略不明确

**问题描述**:
Sprint 0 验证失败后缺少退出策略和 Plan B。

**澄清方案**:

**Go/No-Go 决策矩阵**:

| 验证项 | 通过标准 | 失败后果 | Plan B |
|--------|---------|---------|--------|
| 代码生成质量 | ≥85 分 | 项目暂停 | 降级为模板库 + 人工审查 |
| 多 Agent 延迟 | <30s | 继续但调整目标 | 串行执行 + 缓存优化 |
| 并发性能 | 100 并发无失败 | 继续但降低规模 | 目标降为 100 并发用户 |
| 服务集成 | ≥3 个成功 | 继续但减少范围 | 仅支持核心 5 个服务 |

**三级退出策略**:

**🟢 绿灯（全部通过）**：
- 条件：所有 4 个验证项达标
- 决策：按原计划进入 Sprint 1
- 目标：1000+ 并发用户，完整功能

**🟡 黄灯（部分通过）**：
- 条件：1-3 个验证项未达标
- 决策：调整目标后继续
- 降级方案：
  - 代码质量 75-85 分 → 增加人工审查环节
  - 延迟 30-60s → 调整性能目标为 <60s
  - 并发 50-100 → 目标降为 500 并发用户
  - 服务集成 1-2 个 → 减少集成范围

**🔴 红灯（核心失败）**：
- 条件：代码质量 <75 分或延迟 >60s
- 决策：项目暂停/转型
- 转型方案：
  - 代码质量 <75 分 → 转为半自动化系统（AI 辅助 + 人工编码）
  - 延迟 >60s → 转为异步任务模式（用户提交后邮件通知）
  - 并发 <50 → 转为单租户部署模式

**止损点**:
- ⏱️ **时间止损**：PoC 超过 3 周仍未通过 → 项目暂停
- 💰 **成本止损**：PoC 成本超过 $20,000 → 重新评估
- 🔬 **技术止损**：核心技术假设被证伪 → 项目转型

**结论**: v2.1 增加了完整的 Go/No-Go 决策矩阵、三级退出策略和明确的止损点。

### 11.3 P1-扩缩容：Agent 池扩缩容策略缺失

**问题描述**:
分布式架构设计了 Agent 池，但未说明自动扩缩容触发条件。

**澄清方案**:

**扩容触发条件**:

| 指标 | 扩容阈值 | 持续时间 | 扩容动作 | 目标状态 |
|-----|---------|---------|---------|---------|
| CPU 使用率 | >70% | 5min | +2 Worker 实例 | CPU <60% |
| 队列长度 | >50 | 3min | +3 Worker 实例 | 队列 <30 |
| P95 延迟 | >15s | 5min | +2 Worker 实例 | 延迟 <10s |
| 并发请求数 | >800 | 1min | +5 Worker 实例 | 并发能力 >1000 |

**缩容触发条件**:

| 指标 | 缩容阈值 | 持续时间 | 缩容动作 | 最小保留 |
|-----|---------|---------|---------|---------|
| CPU 使用率 | <30% | 15min | -1 Worker 实例 | 最少 3 个/池 |
| 队列长度 | <10 | 10min | -2 Worker 实例 | 最少 3 个/池 |
| 并发请求数 | <200 | 10min | -3 Worker 实例 | 最少 3 个/池 |

**三级降级机制**:

**L1 降级（轻度过载）**:
- 触发：队列长度 >100
- 动作：启用请求限流（100 req/min → 50 req/min）
- 影响：响应变慢，但功能完整
- 恢复：队列长度 <50 持续 5min

**L2 降级（中度过载）**:
- 触发：队列长度 >200 或 P95 延迟 >30s
- 动作：禁用非核心功能（开源研究、高级优化）
- 影响：仅保留核心流程（需求澄清 → 代码生成 → 部署）
- 恢复：队列长度 <100 且延迟 <20s 持续 10min

**L3 降级（重度过载）**:
- 触发：队列长度 >500 或系统 CPU >90%
- 动作：拒绝新请求，返回 503 + 预计恢复时间
- 影响：系统不可用，显示维护页面
- 恢复：队列长度 <200 且 CPU <70% 持续 15min

**扩缩容配置示例**:

```typescript
const scalingConfig = {
  minInstances: 3,
  maxInstances: 50,
  scaleUpCooldown: 300,
  scaleDownCooldown: 900,
  
  metrics: {
    cpu: { scaleUp: 70, scaleDown: 30 },
    queueLength: { scaleUp: 50, scaleDown: 10 },
    latencyP95: { scaleUp: 15000, scaleDown: 5000 }
  },
  
  pools: {
    analyst: { min: 3, max: 15, priority: 'high' },
    planner: { min: 3, max: 10, priority: 'high' },
    executor: { min: 3, max: 25, priority: 'critical' }
  }
};
```

**结论**: v2.1 增加了完整的扩缩容触发条件、三级降级机制和配置示例。
