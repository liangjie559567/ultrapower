# Ultrapower 改进路线图 - Rough PRD
**版本：** v2.0 (经 5 专家评审修订)
**生成时间：** 2026-03-04
**评审团队：** Product Director, UX Director, Domain Expert, Tech Lead, Critic
**综合评分：** 72.8/100

---

## 执行摘要

经过 5 位专家并行评审，原路线图存在以下核心问题：
1. **工作量严重低估**：原估算 23 周，实际需 54-72 周（+235%）
2. **过度工程化**：EventBus、插件化、分布式 Agent 需求未验证
3. **安全方案不足**：存在 5 种路径遍历绕过向量
4. **性能目标缺乏数据支撑**：未进行 profiling 基线测试

**修订后的核心策略：**
- **Phase 0 (Week 1-3):** 安全加固 Sprint（系统性方案）
- **Phase 1 (Week 4-8):** 性能优化（数据驱动）
- **Phase 2 (Week 9-16):** 质量提升（测试先行）
- **Phase 3 (Week 17-24):** 生态建设（需求验证后）

---

## 1. 优先级重排（基于专家共识）

### P0 - 立即执行（Week 1-8）

#### 1.1 安全加固（系统性方案）
**工作量：** 3 周 → 实际 4-5 周
**负责人：** Security Team
**验收标准：**
- [ ] 通过 OWASP Top 10 安全扫描
- [ ] 路径遍历防护覆盖 5 种绕过向量（URL编码、Unicode、符号链接、空字节、双重编码）
- [ ] 状态文件加密（AES-256-GCM）
- [ ] Hook 输入白名单严格执行
- [ ] 审计日志完整性验证

**Critic 关键建议：**
```typescript
// 系统性路径验证（而非补丁式）
function validatePath(userPath: string, baseDir: string): string {
  // 1. 规范化（处理 Unicode、URL编码）
  const normalized = path.normalize(decodeURIComponent(userPath));
  // 2. 解析符号链接
  const resolved = fs.realpathSync(path.join(baseDir, normalized));
  // 3. 边界检查
  if (!resolved.startsWith(fs.realpathSync(baseDir))) {
    throw new SecurityError('Path traversal detected');
  }
  return resolved;
}
```

#### 1.2 性能优化（数据驱动）
**工作量：** 3 周 → 实际 4 周
**前置条件：** 建立 profiling 基线
**负责人：** Performance Team

**Step 1: Profiling 基线（Week 4）**
- 使用 Chrome DevTools 记录 Hook 执行时间
- 识别真实瓶颈（而非假设）
- 计算 Amdahl's Law 理论上限

**Step 2: Hook 并行化（Week 5-6）**
- 仅并行化无依赖 Hook（需绘制依赖图）
- 添加竞态检测（读-写、写-写）
- 目标：延迟减少 25-30%（而非 40%）

**Tech Lead 警告：**
> 原路线图假设 Hook 完全独立，实际存在隐式依赖。需要完整依赖图分析。

**Step 3: 状态 I/O 优化（Week 7-8）**
- 分级写入策略（关键状态立即写，非关键批量写）
- 添加 WAL（Write-Ahead Log）防止崩溃丢失
- 目标：I/O 次数减少 40%

### P1 - 短期执行（Week 9-16）

#### 1.3 代码质量提升
**工作量：** 2 周 → 实际 4 周
**策略：** 测试先行（而非重构先行）

**Week 9-10: 测试覆盖提升**
- 为现有代码添加测试（50% → 60%）
- 重点：Hooks 系统、状态管理

**Week 11-12: 消除 any 类型**
- 仅在有测试保护的模块中消除 any
- 目标：核心模块 any 使用率 < 5%

**Week 13-16: 循环依赖解决**
- 使用 madge 检测
- 提取共享接口到 core/

#### 1.4 文档体系完善
**工作量：** 2 周
**UX Director 建议：** 优先级提升至 P1

**交付物：**
- 交互式新手引导（Wizard）
- 智能工作流推荐（基于任务类型）
- 任务模板库（常见场景）
- 故障排查手册

### P2 - 中期执行（Week 17-24，需求验证后）

#### 1.5 生态建设（提前到 Q3）
**Product Director 建议：** 配合插件化发布

**Week 17-20: 插件 API 设计**
- 定义插件接口规范
- 沙箱隔离机制（Critic 要求）
- 插件权限模型

**Week 21-24: Agent 市场 MVP**
- 社区插件提交流程
- 插件认证体系
- 开发者文档

## 2. 删除项（专家共识）

### 2.1 EventBus 重构 ❌
**Domain Expert 判断：** 过度工程化
**替代方案：** 依赖注入 + 接口抽象

### 2.2 分布式 Agent ❌
**原因：**
- 需求未验证（当前无用户请求）
- 复杂度爆炸（8 周 → 实际需 24-32 周）
- 缺少脑裂处理和一致性保证

**Tech Lead + Critic 共识：** 暂缓至少 12 个月

### 2.3 可视化编排界面 ❌
**原因：** ROI 不明确，优先级低于文档和 CLI 改进

## 7. 专家评审总结

### 7.1 共识点
- ✅ 安全加固必须优先（P0）
- ✅ 性能优化需要数据驱动
- ✅ 工作量估算过于乐观
- ✅ 分布式 Agent 应删除

### 7.2 分歧点
- ⚠️ 插件化时机：Product 建议 Q3，Domain 建议 Q4
- ⚠️ 可视化界面：UX 建议保留，Tech/Critic 建议删除

### 7.3 仲裁决策
- **插件化：** Q3 启动（配合生态建设）
- **可视化界面：** 延期至 Q4 2026（需求验证后）
- **分布式 Agent：** 删除（至少暂缓 12 个月）
