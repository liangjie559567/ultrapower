# T014 实施计划：状态管理测试覆盖提升

## 目标
- 覆盖率从 45-50% 提升至 >70%
- 补全 WAL 恢复、加密集成、分级写入集成、状态迁移测试

## 阶段划分

### Phase 1: P0 阻塞项（2 人天）
**目标覆盖率**: 45% → 60%

#### 1.1 WAL 恢复机制测试（新增 8 用例）
文件: `src/features/state-manager/__tests__/wal-recovery.test.ts`

测试场景:
- WAL 文件损坏时的回滚
- 部分写入的恢复
- 多条目 WAL 的顺序恢复
- WAL 清理后的状态一致性
- 并发写入时的 WAL 锁定
- WAL 文件不存在时的降级
- 恢复失败时的错误处理
- 恢复后的数据完整性验证

验收标准:
- `wal.ts` 覆盖率 >80%
- 所有恢复路径有测试覆盖

#### 1.2 加密集成测试（新增 6 用例）
文件: `src/features/state-manager/__tests__/encryption-integration.test.ts`

测试场景:
- 加密状态的读写完整流程
- 密钥轮换时的状态迁移
- 加密失败时的降级策略
- 解密失败时的错误处理
- 加密性能边界测试（大文件）
- 加密与 WAL 的协同工作

验收标准:
- `encryption.ts` 与 `index.ts` 集成路径覆盖 >70%
- 密钥管理边界情况全覆盖

## 总计
- **新增测试用例**: 36 个
- **预期覆盖率**: 45% → 75%+
- **总工时**: 5.5 人天
- **并行度**: Phase 1 和 Phase 2 可部分并行（WAL 与分级写入独立）

## 依赖关系
```
Phase 1.1 (WAL) ──┐
                  ├──> Phase 2.1 (分级写入集成)
Phase 1.2 (加密) ──┘

Phase 2.1 ──> Phase 2.2 (状态迁移)
Phase 2.2 ──> Phase 3.1 (清理)
Phase 3.1 ──> Phase 3.2 (边界)
```

## 质量门禁
每个阶段完成后必须:
1. 运行 `npm test -- --coverage`
2. 验证覆盖率达标
3. 所有测试通过
4. 无 TypeScript 错误

## 风险缓解
- **风险**: WAL 恢复测试可能触发真实文件系统操作
  - **缓解**: 使用 `mock-fs` 或临时目录隔离
- **风险**: 加密测试可能暴露密钥
  - **缓解**: 使用测试专用密钥，测试后立即清理
- **风险**: 并发测试可能不稳定
  - **缓解**: 使用 `vitest` 的 `--no-threads` 模式验证
