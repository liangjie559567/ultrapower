# Workflow Patterns

## Team 模式适用场景

**来源**: v5.5.18 P0 修复会话反思 (2026-03-05)

### 适用条件
- ✅ 3+ 个独立任务
- ✅ 每个任务耗时 >30 分钟
- ✅ 依赖关系清晰
- ✅ 问题域明确（可分配专业 agent）

### 不适用条件
- ❌ 高度耦合的顺序任务（如重构 → 测试 → 文档）
- ❌ 任务间共享状态（如全局配置修改）
- ❌ 单个任务 <30 分钟（并行开销不值得）

### 证据
- **v5.5.18 P0 修复**: 6 个任务无依赖冲突，并行度 100%
- **时间节省**: 约 70%（预估 20h → 实际 2h）
- **首次成功率**: 100%（零返工）

### 预期收益
- 相比顺序执行节省 60-70% 时间
- 专业 agent 提升质量
- 清晰的进度可视化

### 风险
- 依赖分析错误导致返工（本次未发生）
- 并行执行的隐藏依赖（需在 planner 阶段检测共享状态）

---

## P0 修复标准流程

**模式**:
```
analyst (识别问题)
  → planner (分解任务 + 依赖分析)
  → team-exec (并行执行，专业 agent)
  → verifier (自动化验证)
  → team-fix (如需修复)
```

**适用条件**: 3+ 个独立 P0 问题，每个有明确问题域

**预期效率**: 相比顺序执行节省 60-70% 时间

**关键成功因素**:
- 依赖分析准确（使用 `addBlockedBy` 显式声明）
- Agent 专业化分工（security-fixer, memory-leak-fixer 等）
- 验证标准可自动化执行

## 文档项目并行化模式

**来源**: v5.5.18 用户使用指南会话 (2026-03-05)

**模式**:
```
专家评审并行 (5 agents)
  → 阶段内任务并行 (Phase 1: T-002 + T-003)
  → 中期验证
  → 阶段内任务并行 (Phase 4: T-007 + T-008)
  → 最终验证
```

**收益**: 99.4% 时间节省（28h → 10min）

**适用场景**: 所有文档类项目
