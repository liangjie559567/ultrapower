# Team 模式最佳实践

**来源**: v5.5.18 P0 修复会话
**置信度**: 高（基于实际数据）
**最后更新**: 2026-03-05

## 适用场景

✅ **使用 Team 模式**：
- 3+ 个独立任务
- 每个任务耗时 >30 分钟
- 依赖关系清晰（可用 DAG 表示）
- 问题域明确（可分配专业 agent）

❌ **不使用 Team 模式**：
- 高度耦合的顺序任务
- 单个任务 <30 分钟
- 任务间共享全局状态

## 预期收益

| 场景 | 时间节省 | 证据 |
|------|---------|------|
| P0 修复（6 任务） | 70% | v5.5.18 (20h → 2h) |
| 文档项目（10 任务） | 99.4% | v5.5.18 用户指南 |
| 代码审查（5 维度） | 80% | v5.5.18 综合审查 |

## 实施步骤

1. **任务分解**：使用 `planner` 分析依赖关系
2. **Agent 分配**：根据问题域选择专业 agent
3. **依赖声明**：使用 `TaskUpdate` 的 `addBlockedBy`
4. **并行执行**：Team 自动调度无依赖任务
5. **验证汇总**：`verifier` 检查所有任务完成

## 常见陷阱

### 陷阱 1: 隐式依赖未声明
**症状**: 任务并行执行时出现冲突
**解决**: 在 `planner` 阶段显式声明所有依赖

### 陷阱 2: Agent 内部轮询
**症状**: 死锁或无限等待
**解决**: 使用 `addBlockedBy` 而非 agent 内部 `while` 循环

### 陷阱 3: 过度并行化
**症状**: 任务切换开销大于收益
**解决**: 仅对耗时 >30 分钟的任务使用并行
