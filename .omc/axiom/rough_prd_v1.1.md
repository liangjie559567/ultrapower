**状态**: 基于用户提供的流程图优化

**版本**: v1.1（整合用户最佳实践）

## 1. 问题陈述

### 1.1 当前痛点

ultrapower 插件生成的代码存在严重质量问题：

* ❌ **语法/类型错误**: 无法通过编译

* ❌ **逻辑错误**: 功能不符合预期

* ❌ **代码质量差**: 不符合最佳实践

* ❌ **功能不完整**: 缺少错误处理、边界情况

### 1.2 根本原因

Agent 缺少完整的开发流程：
1. 不分析现有代码实现
2. 不搜索外部最佳实践
3. 不保证功能完整性
4. 不强制 CI 门禁验证

---

## 2. 解决方案概述

### 2.1 核心理念

**Claude（架构师/评审）+ Codex（执行工程师）双模型协作**

* **Claude**: 需求分析、架构设计、代码评审、测试规划

* **Codex**: 代码实现、文档生成、测试执行

---

## 3. 工作流设计（基于用户最佳实践）

### 3.1 新项目开发流程

#### 一、需求阶段 (Claude)

1. 与 Claude 对话，输出需求文档定稿
2. 让 Claude 搜索竞品对比，补全需求不足
3. 让 Claude 输出技术实现文档

**输出**: `requirements.md` + `tech-design.md`

---

#### 二、开发阶段 (Codex)

1. 将需求文档、技术文档放入项目目录，打开 Codex
2. 让 Codex 先输出目录结构草案确认，再拆分开发周期
3. Codex 按周期开发，每周期结束输出变更摘要

**输出**: 可运行代码 + 周期变更摘要

---

#### 三、优化循环 (Claude ↔ Codex)

1. Codex 根据代码输出功能流程文档
2. Claude 对比需求文档，给出优化清单（注明阻断项 vs 优化项）
3. Codex 按清单修改，更新文档
4. 重复 7-9，直到无优化点

**循环限制**: 最多 5 轮

---

#### 四、测试循环 (Claude ↔ Codex)

1. Claude 给出测试清单
2. Codex 按清单排查，输出测试报告
3. 重复 11-12，直到测试通过

**循环限制**: 最多 10 次

---

### 3.2 老项目改造流程

#### 一、读取现状 (Codex)

1. 让 Codex 读取整个项目，输出功能流程文档
   - **微服务**: 每个服务单独跑一份

**输出**: `feature-flow.md`（每个服务一份）

---

#### 二、生成修改文档 (Claude)

1. 将各服务文档 + 修改需求交给 Claude，输出修改文档
   - **微服务**: 每个服务生成独立修改文档
   - **按功能模块分好结构**

**输出**: `modification-plan.md`（按模块分解）

---

#### 三、拆分开发文档 (Codex)

1. 将修改文档交给 Codex，拆分为每个功能模块独立的开发文档

**输出**: `dev-module-1.md`, `dev-module-2.md`...

---

#### 四、逐模块开发 (Codex ↔ Claude 循环)

**每个功能模块完成以下完整闭环，再进入下一个**:

1. Codex 按开发文档修改该模块
2. Codex 输出该模块的完整功能流程文档
3. Claude 对比原始修改文档，给出优化清单（阻断项 vs 优化项）
4. Codex 按清单优化，更新文档
5. 重复 5-7，直到无优化点
6. Claude 给出测试清单 → Codex 排查 → 循环至测试通过
7. ✅ 该模块完成，进入下一个模块

**循环限制**:

* 优化循环: 最多 5 轮

* 测试循环: 最多 10 次

## 4. 技术实现方案

### 4.1 新增 Skill: `/ccg`

```yaml
触发: 用户启动项目开发
功能:
  - 自动检测项目类型（新/老）
  - 路由到对应工作流
  - 管理文档传递
  - 协调评审循环
  - 实时进度反馈
```

### 4.2 文档模板系统

```
.omc/ccg/
├── requirements.md           # 需求文档
├── tech-design.md            # 技术实现文档
├── feature-flow.md           # 功能流程文档
├── modification-plan.md      # 修改计划（老项目）
├── dev-modules/              # 开发模块文档
│   ├── module-1.md
│   └── module-2.md
├── optimization-list.md      # 优化清单
└── test-checklist.md         # 测试清单
```

### 4.3 关键改进点

1. **周期化开发**: Codex 按周期开发，每周期输出变更摘要
2. **模块化改造**: 老项目按功能模块逐个完成闭环
3. **阻断项标识**: Claude 优化清单明确标注阻断项 vs 优化项
4. **微服务支持**: 每个服务独立生成文档和修改计划

### 4.4 安全加固

```typescript
// 文档路径白名单
const ALLOWED_DOC_TYPES = [
  'requirements', 'tech-design', 'feature-flow',
  'modification-plan', 'optimization-list', 'test-checklist'
];

function getDocPath(type: string): string {
  if (!ALLOWED_DOC_TYPES.includes(type)) {
    throw new Error(`Invalid doc type: ${type}`);
  }
  return `.omc/ccg/${type}.md`;
}

// 用户输入消毒
import { normalizeInput } from './bridge-normalize';
const sanitized = normalizeInput(userInput, {
  allowedFields: ['description', 'features'],
  maxLength: 10000
});
```

### 4.5 降级策略

```yaml
Codex API 不可用:
  - 降级到 Claude 单模型
  - 保留所有流程和提示词

超时处理:
  - 单次调用 > 30s 触发降级
```

## 7. 与 v1.0 的关键差异

| 维度 | v1.0 | v1.1（基于用户流程） |
| ------ | ------ | --------------------- |
| 开发方式 | 一次性开发 | **周期化开发**，每周期输出摘要 |
| 老项目流程 | 按 Phase 执行 | **按模块闭环**，逐个完成 |

| 微服务支持 | 未明确 | **每个服务独立文档** |
| 文档结构 | 6 种文档 | **7 种文档 + 模块目录** |
