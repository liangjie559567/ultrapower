# Critic Review: ultrapower v7.0.0 PRD（第二轮安全评审）

**评审人**: The Critic (批判者)
**评审日期**: 2026-03-10
**PRD 版本**: v7.0.0 Final
**评审重点**: 安全、边界情况、逻辑一致性

---

## 执行摘要

ultrapower v7.0.0 PRD 在整合首轮专家意见后，**安全防护措施显著改善**，但仍存在 **2 个 P0 级安全风险**和 **5 个 P1 级质量风险**。主要问题集中在：并发控制不足、资源耗尽防护缺失、错误恢复机制不完整、测试覆盖不足。建议**有条件通过**，需在 M2 阶段前完成 P0 风险缓解设计。

**评分**: 3.5/5.0

---

## 1. P0 安全风险（必须解决）

### 🔴 P0-1: 并发控制缺失导致状态损坏

**问题描述**:
- PRD 提到"乐观锁（版本号）"和"事务支持"（F2.4），但未定义**并发冲突解决策略**
- 状态迁移（F2.11）只有"备份+回滚"，未处理**迁移过程中的并发写入**
- 缓存系统（F2.9）只有用户隔离，未处理**同一用户的并发请求**

**攻击场景**:
```
时间线：
T1: Agent A 读取状态 v1
T2: Agent B 读取状态 v1
T3: Agent A 写入状态 v2（基于 v1）
T4: Agent B 写入状态 v2'（基于 v1，覆盖 A 的修改）
结果：Agent A 的修改丢失（Lost Update）
```

**影响**:
- 数据完整性破坏（高）
- Team Pipeline 状态不一致（高）
- 用户任务执行结果错误（高）

**缓解要求**:
1. **强制版本检查**: 写入时必须验证版本号，冲突时返回 `409 Conflict`
2. **重试策略**: 冲突后自动重试（最多 3 次），使用指数退避
3. **死锁检测**: 超时机制（30 秒），避免无限等待
4. **并发测试**: 添加 50 并发写入测试（F3.8 已有，需提前到 M2）

**验收标准**:
- ✅ 50 并发写入，冲突率 <5%，零数据丢失
- ✅ 死锁场景自动恢复（30 秒超时）
- ✅ 状态迁移期间拒绝并发写入（返回 503）

---

### 🔴 P0-2: 资源耗尽攻击防护不足

**问题描述**:
- F3.9 "资源限制处理"被放在 M3（2.5 月后），但**资源耗尽是 P0 风险**
- 缺乏**请求速率限制**（Rate Limiting）
- 缺乏**Agent 并发数限制**（可能启动无限 Agent）
- 缺乏**磁盘配额限制**（缓存/日志可能填满磁盘）

**攻击场景**:
```
恶意用户：
1. 启动 100 个并发 Team（每个 10 个 Agent）→ 1000 个 Agent 同时运行
2. 每个 Agent 写入 100MB 日志 → 100GB 磁盘占用
3. 系统 OOM 或磁盘满 → 所有用户服务中断
```

**影响**:
- 拒绝服务（DoS）攻击（高）
- 合法用户无法使用（高）
- 系统崩溃（高）

**缓解要求**:
1. **提前到 M2**: 将 F3.9 从 M3 移到 M2（可靠性阶段）
2. **速率限制**: 每用户每分钟最多 10 个 Agent 启动
3. **并发限制**: 每用户最多 20 个活跃 Agent
4. **磁盘配额**: 每用户最多 1GB 缓存/日志，超限自动清理
5. **内存监控**: 单 Agent 内存 >500MB 触发告警并终止

**验收标准**:
- ✅ 100 并发 Agent 启动请求，系统稳定（拒绝超限请求）
- ✅ 磁盘满场景自动清理（保留最近 7 天数据）
- ✅ 内存泄漏场景自动终止失控 Agent

## 2. P1 质量风险（强烈建议）

### 🟡 P1-1: 熔断器恢复策略不完整

**问题描述**:
- F2.2 提到"渐进式恢复（半开状态限流）"，但未定义：
  - **限流比例**（半开状态允许多少请求？）
  - **恢复条件**（连续成功多少次才完全恢复？）
  - **二次熔断**（恢复后再次失败如何处理？）

**边界情况**:
```
场景：外部 API 间歇性故障（成功率 50%）
问题：半开状态可能反复切换（HALF_OPEN ↔ OPEN），导致抖动
```

**建议**:
1. 半开状态限流 10%（每 10 个请求允许 1 个）
2. 连续成功 5 次才恢复到 CLOSED
3. 半开状态失败立即回到 OPEN，等待时间翻倍（最多 5 分钟）
4. 添加"慢启动"机制（恢复后逐步增加流量）

**验收标准**:
- ✅ 间歇性故障场景不抖动（状态切换 <3 次/分钟）
- ✅ 恢复后性能逐步提升（慢启动）

---

### 🟡 P1-2: 重试机制幂等性验证不足

**问题描述**:
- F2.10 提到"非幂等操作禁止重试"，但**幂等性判断依赖人工声明**（`isIdempotent`）
- 缺乏**运行时幂等性验证**（如何确保声明正确？）
- 缺乏**部分成功处理**（操作执行一半失败如何回滚？）

**边界情况**:
```
场景：write_file 操作
1. 文件写入 50%
2. 网络中断
3. 重试 → 文件内容重复或损坏
```

**建议**:
1. **原子操作**: 写入临时文件 → 验证 → 原子重命名
2. **幂等性测试**: 每个操作必须通过"执行两次结果相同"测试
3. **部分成功检测**: 操作前记录 checksum，失败时对比
4. **补偿事务**: 非幂等操作失败后自动回滚（如删除已创建文件）

**验收标准**:
- ✅ 所有 RETRYABLE_OPERATIONS 通过幂等性测试
- ✅ 部分成功场景自动回滚（零数据损坏）

---

### 🟡 P1-3: 缓存投毒防护不完整

**问题描述**:
- F2.9 提到"缓存键包含 userId + permissions"，但未防护：
  - **时间窗口攻击**（缓存过期前权限变更）
  - **缓存键碰撞**（不同输入生成相同缓存键）
  - **缓存污染**（恶意输入导致错误结果被缓存）

**攻击场景**:
```
时间线：
T1: 用户 A 有权限 P1，查询数据 D → 缓存 (userId=A, perm=P1) → D
T2: 管理员撤销 A 的权限 P1
T3: 用户 A 再次查询 → 命中缓存 → 仍返回 D（权限绕过）
```

**建议**:
1. **权限版本号**: 缓存键包含 `permissionVersion`（权限变更时递增）
2. **缓存键哈希**: 使用加密哈希（SHA-256）避免碰撞
3. **输入验证**: 缓存前验证输入合法性（防止污染）
4. **TTL 上限**: 敏感数据 TTL 最多 5 分钟

**验收标准**:
- ✅ 权限变更后缓存立即失效
- ✅ 100 万次缓存操作零碰撞
- ✅ 恶意输入不被缓存

---

### 🟡 P1-4: 状态迁移回滚机制不完整

**问题描述**:
- F2.11 提到"失败自动回滚"，但未定义：
  - **回滚粒度**（回滚到哪个时间点？）
  - **部分迁移**（如何处理已迁移 50% 的数据？）
  - **回滚失败**（回滚本身失败如何处理？）

**边界情况**:
```
场景：迁移 1000 个状态文件
1. 前 500 个成功迁移到 SQLite
2. 第 501 个失败
3. 回滚 → 如何恢复前 500 个？
4. 回滚失败 → 数据处于不一致状态
```

**建议**:
1. **事务日志**: 记录每个迁移步骤（WAL）
2. **检查点机制**: 每 100 个文件创建检查点
3. **回滚优先级**: 先回滚 SQLite，再恢复文件
4. **双写验证**: 迁移期间双写，对比一致性
5. **人工介入**: 回滚失败时暂停系统，通知管理员

**验收标准**:
- ✅ 迁移中断后自动恢复到最近检查点
- ✅ 回滚失败时系统安全停止（不继续破坏数据）

---

### 🟡 P1-5: 追踪数据脱敏规则不完整

**问题描述**:
- F3.7 提到"敏感信息自动脱敏（API key、password、token）"，但：
  - **脱敏规则不全**（缺少 email、phone、credit card）
  - **上下文泄露**（脱敏后仍可通过上下文推断）
  - **性能开销**（每条日志都正则匹配？）

**边界情况**:
```
日志：User john@example.com logged in with token abc***
问题：email 未脱敏，token 部分可见（可暴力破解）
```

**建议**:
1. **完整规则**: 添加 email、phone、SSN、credit card 脱敏
2. **完全脱敏**: 敏感字段完全替换为 `[REDACTED]`（不保留前缀）
3. **白名单模式**: 只记录明确安全的字段（而非黑名单脱敏）
4. **采样脱敏**: 仅对采样日志（10%）执行脱敏，降低开销

**验收标准**:
- ✅ 100 种敏感信息模式全部脱敏
- ✅ 脱敏后无法通过上下文推断原值
- ✅ 性能开销 <5%

## 3. P2 改进建议（可选）

### 🟢 P2-1: 成本预算控制粒度不足

**问题**: F2.7 只有全局预算，缺乏**分 Agent 预算**和**分阶段预算**

**建议**: 支持 `maxTokensPerAgent` 和 `maxTokensPerPhase`，避免单个 Agent 耗尽预算

---

### 🟢 P2-2: 人工审核节点超时策略不明确

**问题**: F2.8 提到"超时自动拒绝"，但未定义**超时时间**和**通知机制**

**建议**: 默认 5 分钟超时，超时前 1 分钟发送提醒通知

---

### 🟢 P2-3: 模型降级策略过于激进

**问题**: F2.3 降级链 `Opus → Sonnet → Haiku` 可能导致**质量显著下降**

**建议**: 添加"质量阈值"检查，Haiku 结果质量过低时拒绝降级（返回错误）

---

### 🟢 P2-4: 测试覆盖率目标不足

**问题**: 验收标准要求"测试覆盖率 >80%"，但**安全关键代码应 100%**

**建议**: 安全模块（SecureAgentCache、权限验证、状态迁移）要求 100% 覆盖

---

### 🟢 P2-5: 文档缺少安全最佳实践

**问题**: F1.6 快速文档未包含**安全配置指南**

**建议**: 添加"生产环境安全检查清单"（权限配置、日志脱敏、速率限制）

---

## 4. 逻辑一致性检查

### ✅ 通过项

1. **功能依赖清晰**: M1 → M2 → M3 依赖关系合理
2. **验收标准可测量**: 所有指标有明确数值目标
3. **风险评估全面**: 技术/资源/市场风险均覆盖
4. **时间规划合理**: 8 个月含 20% 缓冲

### ⚠️ 不一致项

1. **F3.9 优先级错误**: 资源限制是 P0 风险，不应放在 M3
   - **修正**: 移到 M2，作为 F2.15

2. **F3.8 并发测试滞后**: 并发问题应在 M2 发现，不应等到 M3
   - **修正**: 移到 M2，作为 F2.4 的验收标准

3. **安全指标与功能不匹配**: 验收标准要求"零 P0 漏洞"，但 F2.9/F2.10/F2.11 设计不完整
   - **修正**: 补充 P0-1 和 P0-2 的缓解措施

## 8. 结论

ultrapower v7.0.0 PRD 在整合首轮专家意见后，**整体质量显著提升**，特别是用户体验和基础安全防护。但在**并发控制**和**资源耗尽防护**方面仍存在 P0 级风险，需在 M2 开始前完成设计补充。

**建议批准路径**:
1. 立即启动 M1（性能 + 基础体验）
2. 在 M1 期间完成 P0 风险设计文档
3. M2 开始前进行设计评审
4. M2 实施时同步解决 P1 风险

**预期结果**: 按此路径执行，v7.0.0 可在 8 个月内交付**企业级生产就绪平台**，安全性和可靠性达到行业标准。
