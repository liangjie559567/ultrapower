## 关键问题

### 1. 工作量估算过于乐观（P0）

**问题**: 报告估算 1450 行代码，但未考虑隐藏成本：
- 文件沙箱需要完整的 AST 解析和系统调用拦截（实际 1500+ 行）
- SQLite 方案需要 schema 设计、迁移工具、查询 API（+500 行）
- Worktree 自动化需要处理冲突、清理、错误恢复（+300 行）
- 多模型协作需要重试逻辑、fallback 机制、成本控制（+400 行）

**实际工作量**: 约 3650 行代码 + 测试（2.5x 报告估算）

**影响**: 从"2 周"变为"4-6 周"的项目

### 2. 超时解决方案存在技术债务（P0）

**问题**: `run_in_background: true` 绕过了 Agent 超时，但引入新问题：
- 后台任务失败时无法立即感知（需轮询 TaskOutput）
- 20 个并发限制在批量实验时会成为瓶颈
- 后台任务的日志收集和错误追踪机制不明确

**建议**: 实现专用的 `long_running_task` 工具，支持流式日志和进度回调

### 3. 状态持久化方案未做技术选型（P1）

**问题**: 报告提出 3 个方案但未给出决策依据。

**建议**: 推荐方案 B（SQLite），理由：
- 成熟的并发控制（WAL 模式）
- 原生支持时间序列查询
- 不影响现有 state 工具稳定性
- 可通过 `python_repl` 零依赖实现

### 4. 文件沙箱设计不完整（P0）

**问题**: 报告只提出路径白名单，但未考虑：
- `train.py` 内部可能 `import os; os.system('rm -rf /')`
- Python 的 `eval()`、`exec()`、`__import__()` 绕过
- 子进程创建（`subprocess.Popen`）
- 网络访问控制

**真实工作量**: 800 行 → 2000+ 行（需实现完整的 Python 沙箱）

**替代方案**: 使用 Docker 容器隔离（成本更低，安全性更高）

### 5. 多模型协作的成本控制缺失（P1）

**问题**: 六阶段协作流程每次实验调用 4 次 LLM。

**成本估算**: 50 次实验 × 4 次调用 × $0.02/1K tokens ≈ $40-60/session

**缺失机制**:
- 预算上限控制
- 低质量修改的早期拦截
- Gemini 调用的触发条件

## 改进建议

### 立即修正（P0）

1. **重新评估工作量**: 将 1450 行调整为 3500-4000 行，时间从 2 周调整为 4-6 周

2. **明确状态存储方案**: 选择 SQLite（方案 B），设计 schema 和查询 API

4. **设计成本控制机制**: 每个 session 设置预算上限（如 $50）

### 短期优化（P1）

5. **扩展 Agent 超时配置**: 为 scientist agent 添加可配置超时

6. **实现实验可观测性**: 流式日志输出、状态面板、错误追踪

7. **添加并发实验支持**: 扩展 Ralph Loop 或使用 Team 模式
