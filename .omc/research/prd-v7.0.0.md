# ultrapower v7.0.0 产品需求文档

**版本**: v7.0.0
**制定日期**: 2026-03-10
**状态**: 草案
**负责人**: Product Team

---

## 1. 概述

### 1.1 版本目标

ultrapower v7.0.0 旨在将系统从功能完整的编排框架提升为**企业级生产就绪平台**，重点解决工程化成熟度问题。

**核心目标**:
- 提升系统可靠性至 98%+
- 降低运营成本 30%
- 提升开发效率 50%
- 增强可观测性 100%

### 1.2 核心价值

**对用户**:
- 更稳定的任务执行（自动重试、熔断保护）
- 更低的使用成本（智能降级、结果缓存）
- 更快的问题定位（分布式追踪、性能监控）
- 更好的开发体验（快速测试、模块化架构）

**对团队**:
- 更低的维护成本（统一状态管理、清晰模块边界）
- 更快的迭代速度（测试时间从 34s → 5s）
- 更强的扩展能力（Agent 配置化、工具链标准化）

### 1.3 目标用户

**主要用户**:
- AI 应用开发者（使用 Claude 构建自动化工作流）
- 企业技术团队（需要可靠的 Agent 编排平台）
- 开源贡献者（扩展 Agent/Skill 生态）

**次要用户**:
- 研究人员（探索多 Agent 协作模式）
- 教育机构（教学 AI 编排技术）

## 2. 问题陈述

### 2.1 当前痛点（按优先级）

#### P0: 影响稳定性和性能

**问题 1: 测试套件性能瓶颈**
- **现状**: cli-notify-profile.test.ts 单文件耗时 33 秒，占总测试时间 97%
- **影响**: 开发反馈循环慢，降低测试驱动开发意愿
- **业务价值**: 开发效率直接影响迭代速度和产品竞争力

**问题 2: hooks/bridge.ts 复杂度过高**
- **现状**: 1430 行单文件，承担路由、验证、执行、缓存等多重职责
- **影响**: 启动性能差，维护困难，新增 hook 类型成本高
- **业务价值**: 核心执行路径的性能直接影响用户体验

**问题 3: CLI 启动时间长**
- **现状**: 1651 行 cli/index.ts，加载所有命令即使只用一个
- **影响**: 每次命令执行都需等待 200ms+
- **业务价值**: CLI 响应速度是用户感知的第一印象

#### P1: 影响可靠性和成本

**问题 4: 缺乏自动错误恢复**
- **现状**: 无自动重试、无熔断器、无模型降级
- **影响**: 网络抖动导致任务失败，外部服务故障导致长时间等待
- **业务价值**: 可靠性是企业级应用的基础要求

**问题 5: 状态管理碎片化**
- **现状**: JSON 文件、SQLite、内存缓存混用，无统一抽象
- **影响**: 数据一致性风险，调试困难，迁移成本高
- **业务价值**: 状态管理是系统可靠性的基石

**问题 6: 可观测性不足**
- **现状**: 非标准追踪、无性能监控、无成本追踪
- **影响**: 问题排查困难，性能回归无感知，成本优化缺乏数据
- **业务价值**: 可观测性是生产环境运维的必要条件

**问题 7: 成本优化空间大**
- **现状**: 无结果缓存、无上下文压缩、LSP 工具利用率低
- **影响**: 重复查询浪费成本，代码错误延迟发现
- **业务价值**: 成本优化直接影响用户的使用门槛

#### P2: 影响用户体验

**问题 8: 文档与代码不同步**
- **现状**: 44 处 TODO/FIXME 标记，规范文档滞后
- **影响**: 新贡献者学习曲线陡峭，用户升级遇到意外行为
- **业务价值**: 文档质量影响社区增长

**问题 9: 缺乏可视化调试**
- **现状**: 仅有文本日志，无图形化监控
- **影响**: 学习曲线陡峭，复杂工作流难以理解
- **业务价值**: 可视化降低使用门槛，扩大用户群

### 2.2 影响范围

|------|-----------|---------|---------|

## 3. 解决方案

### 3.1 功能清单（按里程碑）

#### M1: 性能提升版（1-2 周）

**F1.1 测试套件优化**
- 隔离慢速测试（fast vs integration）
- Mock 网络调用（Discord/Telegram API）
- 并行化执行（vitest --threads）
- 目标: 快速测试 <5s，完整测试 <15s

**F1.2 hooks 模块拆分**
- 提取热路径处理器（keyword-detector, pre/post-tool-use）
- 提取生命周期处理器（setup, session-end, subagent-lifecycle）
- 提取验证处理器（permission-request, pre-compact）
- 简化 bridge.ts 为纯路由（<200 行）
- 目标: 启动时间 -30-50%

**F1.3 CLI 命令懒加载**
- 实现 lazyLoadCommand 加载器
- 按需导入命令模块
- 简化 cli/index.ts（<300 行）
- 目标: CLI 启动 -40%

#### M2: 可靠性增强版（1-2 月）

**F2.1 自动重试机制**
- 实现 withRetry 装饰器
- 支持指数退避策略
- 错误分类（可重试/不可重试/需人工）
- 最大重试 3 次
- 目标: 网络抖动自动恢复

**F2.2 熔断器模式**
- 实现 CircuitBreaker 类
- 三态管理（CLOSED/OPEN/HALF_OPEN）
- 连续失败 5 次触发熔断
- 自动恢复探测
- 目标: 避免级联故障

**F2.3 模型自动降级**
- 实现 withModelFallback 函数
- 降级链: Opus → Sonnet → Haiku
- 记录降级事件
- 目标: Opus 限流时自动降级

**F2.4 统一状态管理层**
- 实现 StateManager 接口
- 支持文件系统和 SQLite 后端
- 添加乐观锁（版本号）
- 事务支持
- 目标: 消除状态碎片化

**F2.5 Agent 配置化**
- YAML 格式 Agent 定义
- 分离提示词和配置
- 支持热重载
- 目标: 扩展性 +100%

**F2.6 MCP 服务器统一**
- 实现 createMCPServer 工厂
- 统一 Job 管理逻辑
- 减少代码重复
- 目标: 构建时间 -40%

#### M3: 可观测性版（3-6 月）

**F3.1 OpenTelemetry 追踪**
- 集成 @opentelemetry/api
- 实现 AgentTracer 类
- 支持 Span 嵌套
- 导出到 Console/File/Jaeger
- 目标: 标准化追踪

**F3.2 性能监控面板**
- 记录延迟（P50/P95/P99）
- 统计吞吐量
- 追踪成本（token 消耗）
- 生成性能报告
- 目标: 可观测性 +100%

**F3.3 结果缓存系统**
- 实现 AgentCache 类
- LRU 淘汰策略
- TTL 过期机制
- 缓存键生成（agentType + prompt + context + model）
- 目标: 成本 -30-50%

**F3.4 上下文压缩**
- 自动摘要生成
- 优先级排序
- 增量传递
- 目标: token 消耗 -20-30%

**F3.5 LSP 默认启用**
- executor agent 自动调用 lsp_diagnostics
- 在 exec 阶段发现代码错误
- 减少 fix 循环
- 目标: 错误发现率 +70%

### 3.2 技术方案

#### 架构改进

**当前架构**:
```
用户请求 → CLI → Hook Bridge → [15 种 Hook 处理器] → Agent/Skill
                                                      ↓
                                                   状态文件
```

**目标架构**:
```
用户请求 → CLI → [热路径缓存] → Hook Router → 专用处理器
                                           ↓
                                    [状态管理层] → 持久化
                                           ↓
                                    [重试+熔断] → Agent 执行
                                           ↓
                                    [追踪+监控] → 可观测性
```

**关键改进**:
1. 热路径缓存 → 减少重复加载
2. 统一状态管理 → 消除碎片化
3. 重试+熔断 → 提升可靠性
4. 追踪+监控 → 增强可观测性

#### 技术选型

| 领域 | 技术 | 理由 |
|------|------|------|
| 状态管理 | 文件系统 + SQLite | 简单可靠，支持复杂查询 |
| 可观测性 | OpenTelemetry | 行业标准，生态丰富 |
| 缓存 | LRU + 文件系统 | 轻量级，无额外依赖 |
| 测试 | Vitest + Mock | 快速，TypeScript 友好 |
| 构建 | esbuild + tsc | 快速编译，类型检查 |
